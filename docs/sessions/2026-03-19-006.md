 Critical design decisions (from review)

 Fix #1 — CSV header validation at HTTP boundary, not inside the job

 Validate required columns synchronously inside TaskService::createInvoiceTask() BEFORE
 creating the task. Read only the header row (first line). If description, quantity,
 or unit_price are missing → throw ValidationException → Laravel returns 422.
 The task is never created. Cost: negligible (one fgets call).
 Rationale: "user input error" must fail at the HTTP layer; task failures should mean system
 errors (disk full, OOM, timeout). A Failed task from a typo in CSV headers is wrong semantics.

 Fix #2 — Row-level validation inside the job

 For each row: cast quantity to int (must be > 0), unit_price to float (must be > 0),
 description must be non-empty string. On first invalid row, throw immediately with a
 descriptive message: "Row 3: unit_price must be a positive number."

 Fix #3 — Delete uploaded CSV after PDF is generated

 In the job's finally block: Storage::disk('local')->delete($csvPath).
 The CSV contains financial data and has no reason to persist once the PDF is created.
 Applies on success AND on failure (if CSV was stored before the job errored).

 Fix #4 — Disable dompdf remote resource fetching

 Publish dompdf config and set enable_remote => false explicitly. Prevents a crafted
 CSV description field from triggering outbound HTTP requests via <img src="..."> in
 the rendered Blade template. (Blade escapes {{ }} by default but this is defence in depth.)

 Fix #5 — Stream CSV parsing with fgetcsv()

 Parse rows one at a time in a loop using fgetcsv(). Never load the full CSV into memory.
 Consistent with the cursor() approach used in GenerateReportJob.

 Accepted trade-off

 dompdf's output() returns the full PDF as a string (not a stream). For small invoices
 (tens of rows, ~50KB PDF), this is within PHP's memory_limit without concern. Documented
 as a known constraint — if invoices grew to hundreds of pages, the library choice would
 need revisiting.

 ---
 Security and quality fixes (from senior review)

 Fix A — Floating point arithmetic

 All monetary arithmetic (quantity * unit_price) must use round($qty * $price, 2) per line.
 Grand total is array_sum() of already-rounded line totals.
 Blade template uses number_format($total, 2) for display — never raw float output.
 Rationale: PHP float 0.1 * 3 = 0.30000000000000004. Unacceptable on a financial document.

 Fix B — Row count limit

 GenerateInvoiceJob defines private const MAX_ROWS = 500.
 The fgetcsv() loop increments a counter; if it exceeds MAX_ROWS it throws
 InvalidArgumentException("Invoice exceeds maximum of 500 line items.").
 Task is marked Failed with that message. Rationale: a 5MB CSV with short rows can contain
 50k+ rows; the resulting PDF would be unusable and the job would approach its timeout.

 Fix C — Logo conditional rendering

 Blade template wraps the <img> in @if(file_exists(public_path('images/logo.png'))).
 If the file is absent (new deployment, forgotten file), the PDF renders with company name only —
 no job failure, no cryptic dompdf exception.

 Fix D — Eager-load user in job

 Job guard uses Task::with('user')->find($task->id) instead of a bare find().
 Prevents a second query when $task->user is accessed for the Blade view.

 ---
 Original security fixes (unchanged)

 Fix #1 — CSV header validation at HTTP boundary

 Inside TaskService::createInvoiceTask() BEFORE creating the task: read the header row,
 assert description, quantity, unit_price are present.
 Failure → throw ValidationException → 422. Task never created.

 Fix #2 — Row-level validation inside the job

 Per row: quantity → positive int, unit_price → positive float, description → non-empty.
 On failure: InvalidArgumentException("Row N: {field} must be {rule}.") → task Failed.

 Fix #3 — Delete uploaded CSV in finally block

 Storage::disk('local')->delete($csvPath) in the finally block — runs on success and failure.

 Fix #4 — Disable dompdf remote resource fetching

 Publish dompdf config: enable_remote => false. Logo referenced via local public_path().

 Fix #5 — Stream CSV with fgetcsv()

 Rows parsed one at a time. Full file never loaded into memory.

 Accepted trade-off

 dompdf->output() returns the full PDF string (not a stream). Acceptable for invoices up to
 MAX_ROWS = 500 rows (~50KB PDF). Documented constraint.

 Implementation detail

 GenerateInvoiceJob::handle()

 1. Task::with('user')->find($task->id) guard — exit silently if deleted
 2. Refresh; status guard — exit if not Pending
 3. Mark Processing, progress = 0
 4. try/finally:
   - Open stored CSV, skip header row
   - Row counter; throw if > MAX_ROWS
   - Per row: validate, round($qty * $price, 2), accumulate line items
   - Grand total = array_sum() of line totals
   - Set progress = 50
   - Render Blade view → dompdf → PDF string
   - Store: invoices/{task_uuid}/invoice.pdf
   - Mark Completed, progress = 100, set result_path
   - finally: Storage::disk('local')->delete($csvPath)

 failed(Throwable $e)

 Guard: Task::find(). Update: status = Failed, error_message = $e->getMessage().

 TaskService::createInvoiceTask(User $user, UploadedFile $file): Task

 1. Open file, read header row, assert required columns → ValidationException on failure
 2. Store CSV: uploads/{task_uuid}/{original_name}
 3. Create task: type = InvoiceGeneration, payload = ['file' => $path]
 4. Dispatch GenerateInvoiceJob($task)
 5. Compensating cleanup (matching createAnalysisTask pattern) on any exception

 InvoiceGeneration: filename = invoice-{uuid}.pdf, content_type = application/pdf

● PHP GD — what it is and why it matters here

  GD is a PHP image processing library. It provides functions like imagecreatefrompng(), imagecreatefromjpeg(), and others that let
  PHP read, manipulate, and output raster images (PNG, JPEG, GIF, WebP).

  Why dompdf needs it

  When dompdf converts HTML to PDF, it needs to embed any images referenced in the HTML into the PDF binary. A PDF cannot contain a
  raw PNG file as-is — the image data must be decoded, processed, and re-encoded in a format the PDF spec understands. GD is the
  library dompdf uses to do that decoding. Without it, as soon as dompdf encounters an <img> tag pointing to a PNG, it throws The PHP
   GD extension is required, but is not installed.

  Why our Blade template hit this

  The template conditionally renders a logo with @if(file_exists(public_path('images/logo.png'))). Since public/images/logo.png
  exists in this project, the condition passed, the <img> tag was included in the rendered HTML, dompdf tried to decode the PNG, and
  crashed.

  What changes now that GD is installed

  1. The function_exists('imagecreatefrompng') guard added to the template is no longer strictly necessary — but it is still good
  defensive code. It should stay: if the logo is a PNG and GD is somehow disabled in a future deployment (different server, different
   PHP build), the invoice job would fail silently on the logo rather than crashing entirely.
  2. The tests that run handle() to completion (happy path, 500-row boundary, header-only CSV, float arithmetic) will now produce a
  real PDF with the logo embedded, rather than a PDF without it. The test assertions don't change — they check status, progress, file
   existence, and error messages, not the PDF content itself.

  So in short: GD being installed means the invoice feature now works correctly end-to-end with a logo in the PDF. The guard stays in
   the template as a safety net.
