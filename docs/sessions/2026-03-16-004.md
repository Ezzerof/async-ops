● Good question. Here are my honest thoughts:                                                                                    
                                                                                                                                 
  ---                                                                                                                            
  Can we wrap store + dispatch in DB::transaction?                                                                               
                                                                                                                                 
  Technically yes, but it's the wrong tool for this problem.                                                                     

  DB::transaction protects database consistency — it rolls back DB writes if something throws. But the failure modes here are:

  1. $file->store() fails — this is a filesystem operation, not a DB write. A rollback won't undo files already written to disk.
  You'd have orphaned files with no task pointing to them.
  2. Bus::batch()->dispatch() fails — the batch is dispatched to the queue (Redis/DB). A DB transaction rollback would prevent
  the batch DB record being committed if you're using the database queue driver, but it has no effect on Redis. The behaviour
  depends on your queue driver — fragile.
  3. The task Task::create() fails — this is a DB write, so a transaction correctly rolls it back.

  So a transaction only partially solves the problem and gives a false sense of safety. What you actually need is compensating
  logic: if any step fails, explicitly undo the previous steps (delete the task, delete stored files). That's the pattern for
  operations that span multiple infrastructure boundaries (DB + filesystem + queue).

  ---
  Does wrapping it in a transaction violate SRP?

  Yes, and it makes it worse.

  If you put DB::transaction in the controller, the controller now knows:
  - How HTTP input maps to domain objects
  - How files are stored
  - How jobs are built
  - That the operation needs transactional safety

  That's four responsibilities. The controller becomes an orchestrator that knows too much about infrastructure.

  The cleaner answer is: the transaction (or compensating logic) belongs in TaskService, because that's where the multi-step
  orchestration already lives. The controller's only job is to hand off validated input and return a response.

  ---
  The honest trade-off

  For a production system: compensating logic in TaskService is the right call — explicit, testable, and doesn't couple
  infrastructure concerns to the controller.

  For a demo/assessment: a try/catch with $task->delete() in the controller is pragmatic and good enough. The important thing is
  that the failure case is handled somewhere — right now it isn't.

  ---
  My recommendation: keep the transaction out, add compensating logic inside TaskService (since it already owns task creation and
   dispatch), and keep the controller blind to those details. Want to proceed with that approach when we get to fixing?

  ---
  SOLID Violations

  1. SRP — hydrateBatchProgress is business logic inside the controller

  The controller now knows: how to look up a batch by ID, how processedJobs() and totalJobs relate to percentage, and that
  batch-based tasks behave differently from report tasks. This is orchestration logic that belongs in TaskService. If the
  progress formula changes (e.g. weighted by file size), the controller has to change — that's a violation.

  Fix: Move to TaskService::enrichWithBatchProgress(Task): void and keep the controller blind to the Bus facade entirely.

  ---
  2. DIP — Bus facade coupled directly to the controller

  Bus::findBatch() is called directly in the controller, creating a dependency on Laravel's queue infrastructure. Controllers
  should depend on abstractions (services), not concrete infrastructure facades. This also makes show() harder to unit-test
  without mocking the facade.

  ---
  3. OCP — download() is not open for extension

  download() hardcodes text/csv and report-{uuid}.csv for all tasks. Adding conversion task support (step 12) will require
  branching if/else logic directly in this method rather than extending via a clean abstraction. This is the OCP violation step
  12 will expose.

  ---
  Vulnerabilities

  4. Path traversal in download()

  $task->result_path is read from the DB and passed directly to Storage::disk('local')->download() with no boundary check. If
  result_path were ever set to a crafted path (e.g. ../../.env), it would serve that file. The risk is low since result_path is
  system-set, but there's no defence-in-depth guard like the realpath check already applied in TaskService::zipOutputFiles().

  ---
  Bugs

  5. download() serves wrong Content-Type and filename for conversion tasks

  'report-' . $task->uuid . '.csv' and 'Content-Type' => 'text/csv' are hardcoded. A completed conversion task (e.g. output.json

   the file. This is a correctness bug that step 12 must address.

  TaskPolicy::download() only checks result_path !== null. It does not verify the file exists on disk. If the file was deleted or
