 File Conversion Feature — Design Trade-off Analysis
                                                                                   
 ▎ Context: No code generated. This document analyses design decisions for the   
 async file conversion feature before implementation begins. The goal is to
 surface trade-offs, compare realistic alternatives, and settle on a recommended
 approach that fits the project's demo/assessment context.

 ---
 Decision 1: Batching Strategy

 How should jobs be dispatched when a user uploads one or more files?

 Option A — Always Bus::batch(), even for a single file

 - How it works: Every conversion request goes through Bus::batch([ConvertFileJob,
  ...]), regardless of file count.
 - Advantages: Single, uniform code path; progress is free via $batch->progress();
  then/catch callbacks handle completion and failure cleanly; easy to demonstrate
 batching at any time.
 - Disadvantages: Overhead of writing a job_batches record even for a single file;
  slightly more wiring in TaskService.
 - Simplicity: High — one code path to write, read, and test.
 - Scalability: Good — batch records are lightweight; Redis handles queue churn
 well.

 Option B — Direct dispatch for 1 file, Bus::batch() for multiple

 - How it works: TaskService branches on file count: dispatch(new ConvertFileJob)
 vs Bus::batch([...]).
 - Advantages: Less overhead for the common single-file case.
 - Disadvantages: Two code paths to maintain; progress derivation in show() must
 branch too (batch vs direct job); harder to test uniformly; complexity grows
 disproportionately for the benefit.
 - Simplicity: Low — dual paths throughout.
 - Scalability: Marginally better for single-file load, but negligible at this
 scale.

 Option C — Chunked batches (e.g., max 5 jobs per batch)

 - How it works: Split files across multiple batches if count exceeds a threshold.
 - Advantages: Handles arbitrarily large uploads without overloading a single
 batch.
 - Disadvantages: Completely overkill for the stated 10-file max; complicates
 batch_id tracking (multiple IDs per task); adds orchestration complexity.
 - Simplicity: Very low.
 - Scalability: High, but unnecessary here.

 Recommendation

 Option A. The 10-file cap means the overhead of a single job_batches record per
 request is irrelevant. The uniform code path and free batch progress are worth
 more than micro-optimising the single-file case.

 ---
 Decision 2: Progress Tracking

 How should the progress integer on the Task be kept up to date while jobs run?

 Option A — Derive from Bus::findBatch($batchId)->progress()

 - How it works: batch_id is stored in task->payload. TaskController::show()
 checks if type === 'file_conversion', then queries the batch and returns its
 progress directly.
 - Advantages: Zero DB writes from jobs themselves; progress is accurate (batch
 tracks processed/total jobs); no race conditions; clean separation of concerns.
 - Disadvantages: show() gains a conditional branch; batch records can be pruned
 by php artisan queue:prune-batches (progress disappears after batch is gone —
 harmless for Completed tasks since progress=100 was set in then callback); one
 extra query per show() call.
 - Simplicity: Moderate — extra branch in show(), but job code stays simple.
 - Scalability: Excellent — no chatty writes, batch API scales with the queue
 driver.

 Option B — Each ConvertFileJob writes to task->progress on completion

 - How it works: Each job loads the Task and increments progress by 100 /
 total_files.
 - Advantages: Consistent with the existing GenerateReportJob pattern; no
 dependency on batch state.
 - Disadvantages: Requires Task reference in each job (constructor injection);
 concurrent writes to the same row create a race — without a lock, two jobs
 completing simultaneously can clobber each other's progress values; N DB writes
 per batch instead of 0.
 - Simplicity: Familiar pattern, but the concurrency hazard adds hidden
 complexity.
 - Scalability: Poor — N concurrent updates to a single row is the classic hot-row
  problem.

 Option C — Set progress only at batch completion (0 → 100)

 - How it works: Progress stays 0 until then callback fires and sets it to 100.
 - Advantages: Simplest possible implementation; no race conditions; no extra
 queries.
 - Disadvantages: No visible intermediate progress for the client — the feature
 becomes polling-with-no-feedback, which defeats the purpose of the progress
 field.
 - Simplicity: Very high.
 - Scalability: Best, but at the cost of feature value.

 Recommendation

 Option A. The batch API was designed exactly for this pattern. The conditional
 branch in show() is contained and testable. Option B's concurrency hazard is a
 real bug waiting to happen when multiple files finish simultaneously.

 ---
 Decision 3: Target Format Scope

 Should target_format apply to the whole request, or can each file specify its
 own?

 Option A — Single target_format per request (all files get the same conversion)

 - How it works: POST /api/conversions accepts files[] + one target_format field.
 Every file in the batch is converted to the same output format.
 - Advantages: Simple request shape; straightforward validation; job constructor
 only needs one format value; no mapping/pairing complexity.
 - Disadvantages: Cannot mix conversions (e.g., CSV→JSON and XML→CSV in the same
 request).
 - Simplicity: High.
 - Scalability: Fine — the limitation is a UX constraint, not a technical one.

 Option B — Per-file format mapping (files[0][file], files[0][target_format], …)

 - How it works: Multipart body carries an array of objects, each pairing a file
 with its desired output format.
 - Advantages: Maximum flexibility; each file has its own conversion intent.
 - Disadvantages: Complex multipart validation (PHP/Laravel doesn't handle nested
 multipart cleanly); StoreConversionRequest rules become significantly more
 complex; job setup requires index-aligned pairing; harder to call from clients;
 payload storage becomes a list of {filename, target_format} pairs.
 - Simplicity: Low.
 - Scalability: Technically fine, but the added complexity is not justified for
 this demo.

 Recommendation

 Option A. Per-request format is a sensible constraint for a demo. If per-file
 control were genuinely needed, a better pattern would be separate requests — not
 a complex nested multipart body.

 ---
 Decision 4: Result Delivery

 How should converted files be served to the client?

 Option A — Conditional: single file for 1 upload, zip for multiple

 - How it works: then callback checks file count. Single → set result_path to the
 converted file directly. Multiple → zip all outputs, set result_path to the zip.
 - Advantages: Single-file download is a clean direct stream with no unnecessary
 compression; multi-file is ergonomic (one download); Content-Type and filename
 semantics are natural.
 - Disadvantages: then callback has a branch; TaskController::download() may need
 to adjust Content-Type/filename based on type (text/csv vs application/zip).
 - Simplicity: Moderate — two branches, but both are simple and well-isolated.
 - Scalability: Good.

 Option B — Always zip, even for a single file

 - How it works: All outputs are zipped regardless of count.
 - Advantages: Completely uniform download path; no branching in then or
 download().
 - Disadvantages: Unnecessary overhead to zip one file; client must unzip to get a
  single file — poor ergonomics; result_path always points to a zip which makes
 the path semantics odd for one-file tasks.
 - Simplicity: High in code, low in UX.
 - Scalability: Irrelevant.

 Option C — Serve files individually (no zip at all)

 - How it works: result_path stores only the first converted file, or the task
 returns a list of paths.
 - Advantages: No ZipArchive dependency; simpler storage.
 - Disadvantages: Multi-file downloads require multiple client requests, which
 defeats the purpose of a batch endpoint; the existing download() endpoint returns
  one file — extending it to return a list would require changing the response
 contract significantly.
 - Simplicity: Low overall.
 - Scalability: Not applicable.

 Recommendation

 Option A. The branch is small and localised. Forcing clients to unzip a single
 file (Option B) is unnecessary friction. Option A delivers the most natural
 experience for both cases.

 ---
 Decision 5: Failure Handling

 What happens when one or more jobs in the batch fail?

 Option A — All-or-nothing: any failure marks the whole Task as Failed

 - How it works: Bus::batch()->catch() callback sets task to Failed and stores the
  error message. Individual ConvertFileJob::failed() (Laravel's job-level failure
 hook) can set task to Failed directly if the job exhausts retries.
 - Advantages: Matches the existing TaskStatus state machine exactly (Pending →
 Processing → Completed | Failed); clear, binary outcome; simple for clients to
 reason about; no partial result handling needed.
 - Disadvantages: One bad file loses all successfully converted files; client must
  re-upload all files to retry.
 - Simplicity: High — existing state machine works unchanged.
 - Scalability: Fine.

 Option B — Partial success: complete with whatever succeeded, report failures in
 payload

 - How it works: Batch catch does not fail the task. Each job writes its outcome
 to the task's payload. then zips only the successful outputs and marks the task
 Completed with a warnings payload field.
 - Advantages: More resilient; user gets partial output.
 - Disadvantages: TaskStatus.Completed with partial output is semantically
 misleading; adds payload schema complexity; show() and download() need to handle
 partial states; test coverage doubles; the progress column no longer maps cleanly
  to batch progress.
 - Simplicity: Low — fundamental change to the state machine semantics.
 - Scalability: Not meaningfully better.

 Option C — Per-job retries (3 attempts with backoff) before all-or-nothing
 failure

 - How it works: Option A, but each job has $tries = 3 and backoff(). Transient
 failures (network, temp I/O) auto-retry before propagating.
 - Advantages: Handles transient failures gracefully; only truly unrecoverable
 errors fail the batch; combines well with Option A semantics.
 - Disadvantages: None significant — this is purely additive.
 - Simplicity: High — just adds $tries and backoff() to the job (mirrors
 GenerateReportJob).
 - Scalability: Good.

 Recommendation

 Option A + Option C combined. All-or-nothing outcome keeps the state machine
 clean. Per-job retries (matching GenerateReportJob pattern) handle transient
 errors before they escalate to batch failure.

 ---
 Decision 6: Controller Structure

 Where should the "create a conversion task" logic live?

 Option A — Dedicated ConversionController (handles upload + Task creation only)

 - How it works: POST /api/conversions → ConversionController::store(). Status
 polling and download still go through existing TaskController::show() and
 TaskController::download().
 - Advantages: TaskController stays focused; ConversionController has its own
 FormRequest tailored to multipart/files; concerns are separated; follows the
 existing pattern (one controller per resource/action group).
 - Disadvantages: The conversion entry point has a different base URL than the
 polling/download URLs — but this is a natural REST separation (create resource vs
  read resource).
 - Simplicity: High — clear responsibility boundaries.
 - Scalability: Good — new action types get their own controller without bloating
 TaskController.

 Option B — Add a conversion() method to TaskController

 - How it works: POST /api/tasks/conversions → TaskController::conversion().
 - Advantages: All task operations in one controller.
 - Disadvantages: TaskController grows with each new task type; form request
 handling for multipart inside a JSON-oriented controller is awkward; violates
 single-responsibility.
 - Simplicity: Low — controller becomes a catch-all.
 - Scalability: Poor — every new task type adds to one file.

 Option C — Unified POST /api/tasks (extend StoreTaskRequest to handle files)

 - How it works: Reuse the existing task creation endpoint. type=file_conversion
 triggers different handling in TaskService.
 - Advantages: One entry point for all task creation.
 - Disadvantages: application/json and multipart/form-data are fundamentally
 different content types — a single endpoint cannot cleanly handle both without
 messy content-type detection; StoreTaskRequest validation rules become deeply
 conditional.
 - Simplicity: Very low.
 - Scalability: Poor — conflating resource types.

 Recommendation

 Option A. Dedicated controller per action type is the cleaner pattern. The fact
 that polling/download still uses TaskController is correct REST behaviour — those
  operate on the Task resource, not the conversion action.

 ConversionFormat Enum — Motivated Recommendation

 Recommendation: Yes — a backed string Enum for target_format.

 The project already uses TaskStatus: string as a backed string enum. Using the
 same pattern for ConversionFormat is consistent, predictable, and idiomatic for
 this codebase.

 Why an Enum genuinely earns its keep here:

 1. Validation becomes declarative — StoreConversionRequest uses
 Rule::enum(ConversionFormat::class) instead of a hardcoded ['json', 'csv', 'xml',
  'simulated'] array. The Enum is the single source of truth; add a case and
 validation automatically accepts it.
 2. The Enum can carry behaviour — Unlike a string constant or config array, an
 Enum can have methods. This is where it pays off:
   - outputExtension(): string — maps Json→'json', Csv→'csv', Xml→'xml',
 Simulated→ computed from source extension. The service never constructs output
 filenames with raw string logic.
   - isRealConversion(string $sourceExtension): bool — the service calls this to
 decide real-path vs stub-path. Routing logic lives on the Enum, not scattered in
 conditionals.
 3. Type safety at the service boundary —
 FileConversionService::convert(UploadedFile $file, ConversionFormat
 $targetFormat) cannot be called with an arbitrary string. Once
 StoreConversionRequest casts the input to the Enum, every downstream consumer is
 guaranteed a known value.

 Why NOT to Enum the source/input formats:

 Source format is inferred from the uploaded file's extension — it is not
 user-provided. There is nothing to validate against an Enum. The
 FileConversionService reads the extension via
 pathinfo($file->getClientOriginalName(), PATHINFO_EXTENSION) and routes with a
 match expression. An Enum here would require updating two places (Enum + mimes
 rule) whenever a new fake format is added, rather than just the validation rule.
 A plain mimes rule in StoreConversionRequest is sufficient and more maintainable.

 ConversionFormat cases:
 - Json = 'json'
 - Csv = 'csv'
 - Xml = 'xml'
 - Simulated = 'simulated' — explicit stub request, useful for demo purposes

 - Bus::batch() used for every request, regardless of file count — uniform code

 - In TaskController::show(): if task->type === 'file_conversion' and

 GenerateReportJob pattern)

 File Conversion Feature — Step-by-Step Implementation Plan

 Context

 Extend async-ops with a file_conversion task type. Users upload 1–10 files, select a target format, and receive converted
 output via the existing task polling + download flow. Uses Bus::batch() for all requests; progress is derived from the batch
 API; failures are all-or-nothing with per-job retries.

 Step 1 — ConversionFormat Enum

 What to build: app/Enums/ConversionFormat.php — a backed string enum with cases for all accepted output formats (Json, Csv,
 Xml, Pdf, Docx, Xlsx, Yaml, Txt) and two methods: isRealConversion(string $sourceExt): bool and outputExtension(): string.

 Purpose: This is the single source of truth for all accepted target formats, the routing decision (real vs stub), and output

 Laravel components: Enum (PHP 8.1 backed enum, matching TaskStatus style). No Laravel-specific dependencies.

 Why first: Everything else references this Enum. Building it first means validation, service routing, and job logic can all be

 Laravel components: Storage facade (reading/writing files), ConversionFormat enum for routing. Otherwise plain PHP: str_getcsv,

● How match works                                                                                                                
                                                                                                                                 
  match is like a cleaner switch. It compares a value against arms and returns the result of the first one that matches. It's an 
  expression — it returns a value directly.                                                                                      
                                                                                                                                 
  $status = 'active';                                                                                                            

  $label = match($status) {
      'active'   => 'User is active',
      'banned'   => 'User is banned',
      default    => 'Unknown status',
  };

  // $label = "User is active"

  match(true) is a special trick. Instead of comparing one value against fixed options, each arm is a full boolean expression and
   the first one that evaluates to true wins:

  $age = 25;

  $group = match(true) {
      $age < 18          => 'minor',
      $age >= 18 && $age < 65 => 'adult',
      $age >= 65         => 'senior',
  };

  // $group = "adult"

  This is exactly what realConvert() uses — because we need to check two things at once (the source extension and the target
  format).

  ---
  csvToJson line by line

  $content = Storage::get($sourcePath);
  Reads the entire CSV file from disk into a string. $content might look like:
  name,age,city
  Alice,30,London
  Bob,25,Paris

  ---
  $lines = explode("\n", trim($content));
  trim() strips any trailing newline at the end of the file. explode("\n") splits the string into an array of lines:
  ['name,age,city', 'Alice,30,London', 'Bob,25,Paris']

  ---
  $headers = str_getcsv(array_shift($lines));
  array_shift() removes and returns the first element from $lines — the header row. After this, $lines only contains data rows.
  str_getcsv() parses that CSV string into an array:
  $headers = ['name', 'age', 'city']

  ---

  foreach ($lines as $line) {
      if ($line === '') {
          continue;
      }
      $rows[] = array_combine($headers, str_getcsv($line));
  }
  Loops over each remaining data line. Skips empty lines (common at end of files). For each line:
  - str_getcsv($line) parses it → ['Alice', '30', 'London']
  - array_combine($headers, ...) zips headers and values together → ['name' => 'Alice', 'age' => '30', 'city' => 'London']
  - That array is appended to $rows

  After the loop: $rows = [['name'=>'Alice', 'age'=>'30', 'city'=>'London'], ['name'=>'Bob', ...]]

  ---

  conversions/abc123/data.json.

  JSON_UNESCAPED_UNICODE keeps characters like é, ü, 中 as-is instead of \u00e9.

  ---

  Returns the path of the newly written file so the job knows where to find it.

  1. XXE (XML External Entity) injection — line 91                                                                               
                                                                                                                                 
  $xml = simplexml_load_string(Storage::get($sourcePath));                                                                       
                                                                                                                                 
  By default, simplexml_load_string (and libxml underneath it) will process external entities declared in XML. An attacker could
  upload a file like:

  <?xml version="1.0"?>
  <!DOCTYPE foo [
    <!ENTITY xxe SYSTEM "file:///etc/passwd">
  ]>
  <root>&xxe;</root>

  This causes PHP to read /etc/passwd (or any file the process can access) and embed its contents in the output. The fix is to

  libxml_disable_entity_loader(true);  // PHP < 8.0
  // or pass LIBXML_NOENT | LIBXML_DTDLOAD flags carefully

  In PHP 8.0+ external entity loading is disabled by default, but it's still worth being explicit with LIBXML_NONET to block
