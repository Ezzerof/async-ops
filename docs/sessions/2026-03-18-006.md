● Columnar Stores — Plain English                                                                                                
                                                                                                                               
  The Problem with Row-Based Storage                                                                                             
   
  A normal relational database (MySQL, PostgreSQL) stores data row by row on disk:                                               
                                                                                                                               
  Row 1: [Alice, alice@email.com, 120.50, 2024-01-01]                                                                            
  Row 2: [Bob,   bob@email.com,   85.00,  2024-01-02]   
  Row 3: [Carol, carol@email.com, 200.00, 2024-01-03]                                                                            
                                                                                                                                 
  When you ask "what is the SUM of the amount column across 1 million rows?", the database has to read every full row — name,    
  email, date, and amount — just to get to the one number it needs. Most of that data is loaded into memory and immediately      
  thrown away.                                                                                                                   
                                                                                                                               
  ---
  The Columnar Idea
                   
  A columnar store flips this. It stores data column by column on disk:
                                                                                                                                 
  name column:   [Alice, Bob, Carol, ...]
  email column:  [alice@..., bob@..., carol@..., ...]                                                                            
  amount column: [120.50, 85.00, 200.00, ...]                                                                                    
  date column:   [2024-01-01, 2024-01-02, ...]                                                                                   
                                                                                                                                 
  Now when you ask for SUM(amount), the database reads only the amount column — a tight sequence of numbers, nothing else. This  
  is dramatically faster for analytical queries because:                                                                         
                                                                                                                                 
  1. Less I/O — you only read the columns you ask for                                                                            
  2. Better compression — a column of numbers compresses far better than mixed-type rows (similar values are adjacent on disk)
  3. CPU-friendly — modern CPUs can process arrays of same-type values using SIMD instructions (doing 8 additions at once instead
   of 1)                                                                                                                         
                                                                                                                                 
  The trade-off: inserting or updating a single row is slower, because you have to touch multiple column files instead of one    
  contiguous row.                                                                                                              
                                                                                                                                 
  ---                                                                                                                          
  Parquet Files
                                                                                                                                 
  Parquet is a file format (not a database). Think of it as a very smart way to save a spreadsheet to disk using the columnar
  layout.                                                                                                                        
                                                                                                                               
  - You write a Parquet file once, read it many times                                                                            
  - Tools like Spark, DuckDB, Pandas, and BigQuery can all read the same Parquet file                                          
  - It is compressed by default and includes metadata (min/max values per column block) so queries can skip entire sections of   
  the file without reading them                                                                                                  
  - Common in data pipelines: "export data as Parquet, store in S3, query with whatever tool you want"                           
                                                                                                                                 
  For the CSV import use case: instead of inserting rows into MySQL, you would parse the CSV, write it as a Parquet file to      
  disk/S3, and the analysis service would read it directly. No DB growth, much faster aggregations.                              
                                                                                                                                 
  ---                                                                                                                          
  ClickHouse
            
  ClickHouse is a full database built around the columnar layout. You query it with SQL (mostly), but it is designed specifically
   for analytical workloads — counting, summing, averaging across billions of rows.                                              
  
  Key characteristics:                                                                                                           
  - Inserts are batched (not great for one-row-at-a-time writes, excellent for bulk inserts)                                   
  - Reads for aggregations are extremely fast — 10–100x faster than MySQL on analytical queries                                  
  - Not designed for frequent updates or deletes (it is append-friendly)                       
  - Used by companies like Cloudflare, Uber, and Spotify for their event/log analytics                                           
                                                                                                                                 
  For the CSV import use case: you would insert all CSV rows into ClickHouse once, then run SELECT MIN(amount), MAX(amount),     
  SUM(amount) and get the answer in milliseconds even on 50 million rows.                                                        
                                                                                                                                 
  ---                                                                                                                            
  Summary Table                                                                                                                
                                                                                                                                 
  ┌───────────────────────────┬───────────────────────────┬───────────────────────────────┬──────────────────────────────────┐
  │                           │     MySQL (csv_rows)      │         Parquet file          │            ClickHouse            │   
  ├───────────────────────────┼───────────────────────────┼───────────────────────────────┼──────────────────────────────────┤   
  │ What it is                │ Relational DB table       │ File format                   │ Analytical DB                    │   
  ├───────────────────────────┼───────────────────────────┼───────────────────────────────┼──────────────────────────────────┤   
  │ Best for                  │ Transactions, small reads │ Archiving, portable datasets  │ Fast aggregations at scale       │   
  ├───────────────────────────┼───────────────────────────┼───────────────────────────────┼──────────────────────────────────┤   
  │ SUM on 1M rows            │ Slow (reads all row data) │ Fast (reads only that column) │ Very fast (columnar + optimised) │   
  ├───────────────────────────┼───────────────────────────┼───────────────────────────────┼──────────────────────────────────┤   
  │ Updates/deletes           │ Easy                      │ Not supported                 │ Difficult                        │

 AsyncOps needs a CSV Import feature that feeds the existing CSV Data Analysis and future downstream features (PDF generation,
 bulk email). The goal is a cohesive data pipeline, not a standalone demo.

 Revised from initial design: The csv_rows table approach was dropped. Storing every row in MySQL JSON columns creates an
 unbounded table, poor query performance on JSON fields, and orphaned-row risk on failure. The hybrid approach is used instead:
 the validated CSV file lives on disk permanently, analysis results are computed once and cached in a dedicated table,
 downstream features stream the file directly.

 ---
 Storage Strategy: Hybrid (File on Disk + Cached Analysis Results)

 ┌────────────────────────────────┬─────────────────────────────────────────────────────────────────────────────────┐
 │            Concern             │                                    Decision                                     │
 ├────────────────────────────────┼─────────────────────────────────────────────────────────────────────────────────┤
 │ Raw uploaded file              │ Stored temporarily under uploads/{task_uuid}/, deleted after validation         │
 ├────────────────────────────────┼─────────────────────────────────────────────────────────────────────────────────┤
 │ Validated CSV file             │ Stored permanently under imports/{task_uuid}/filename.csv                       │
 ├────────────────────────────────┼─────────────────────────────────────────────────────────────────────────────────┤
 │ Analysis results               │ Computed once by AnalyseCsvJob, stored in csv_analysis_results table            │
 ├────────────────────────────────┼─────────────────────────────────────────────────────────────────────────────────┤
 │ Downstream access (PDF, email) │ Stream directly from imports/{task_uuid}/filename.csv via Storage::readStream() │
 └────────────────────────────────┴─────────────────────────────────────────────────────────────────────────────────┘

 No csv_rows table. No unbounded growth. No JSON query performance issues.

 API Surface

 POST   /api/imports                          → CsvImportController@store
 GET    /api/imports/{import}                 → CsvImportController@show
 DELETE /api/imports/{import}                 → CsvImportController@destroy
 POST   /api/imports/{import}/analyse         → CsvImportController@analyse
 GET    /api/imports/{import}/analysis        → CsvImportController@analysisResults
 GET    /api/tasks/{task}                     → TaskController@show (unchanged — poll progress)

 All routes under auth:sanctum middleware.

 Flow:
 1. POST /api/imports → returns Task (201). Client polls GET /api/tasks/{uuid} until completed.
 2. POST /api/imports/{import}/analyse → triggers AnalyseCsvJob, returns a new Task. Client polls until completed.
 3. GET /api/imports/{import}/analysis → returns cached csv_analysis_results rows. Fast read, no computation.
 4. DELETE /api/imports/{import} → deletes import record, analysis results (cascade), and the file from disk.

 ---
 Validation: Two-Phase (mirrors existing pattern)

 StoreImportRequest (sync, returns 422):
 - File present, single file
 - MIME: text/csv or application/csv
 - Extension: csv
 - Max size: 10 MB

 ImportCsvJob (async, marks task failed with message):
 - File still exists on disk
 - File parseable as CSV via fgetcsv
 - Header row present, at least one column, no duplicate header names
 - Every row's column count matches header count (malformed row = fail whole job — atomic)

 ---
 Job 1: ImportCsvJob (single job, not batched)

 Validates and stores the file. No row insertion. Mirrors GenerateReportJob structure.

 Step-by-step:
 1. Guard: task exists and status = pending (idempotency — same as all existing jobs)
 2. task.status = processing, progress = 0
 3. Open uploaded file via fgetcsv, read and validate headers (non-empty, no duplicates)
 4. Stream all rows, validate column count per row, count total rows
 5. Progress updates at each ~10% threshold (mirrors GenerateReportJob pattern)
 6. Copy validated file from uploads/{task_uuid}/ to imports/{task_uuid}/filename.csv via Storage::copy()
 7. CsvImport::create([task_id, user_id, original_filename, file_path, headers, row_count, status=completed])
 8. Delete raw uploaded file: Storage::delete('uploads/{task_uuid}/')
 9. task.status = completed, progress = 100
 10. failed(Throwable $e): mark task as failed with error message, delete uploaded file if it exists

 Config: $tries = 3, $timeout = 120, backoff(): [10, 30, 60]

 Note: A single CsvImport::create() at the end is one atomic DB write — no transaction needed here. If the job fails before step
  7, there is no partial DB state to clean up.

 ---
 Job 2: AnalyseCsvJob (single job, not batched)

 Triggered on demand by POST /api/imports/{import}/analyse. Computes statistics and caches them.

 Step-by-step:
 1. Guard: task exists and status = pending
 2. Guard: import exists and status = completed (cannot analyse a failed import)
 3. task.status = processing, progress = 0
 4. Open imports/{task_uuid}/filename.csv via Storage::readStream() + fgetcsv
 5. Stream rows, accumulate per-column numeric values (cast, skip non-numeric)
 6. Progress updates per ~10% of row_count
 7. Wrap result inserts in DB::transaction():
   - Delete any existing results for this import (re-analysis support)
   - CsvAnalysisResult::insert($results) — one row per column
 8. task.status = completed, progress = 100
 9. failed(Throwable $e): mark task as failed

 Config: $tries = 3, $timeout = 120, backoff(): [10, 30, 60]

 Transaction wrapping here is important: if inserting results for 10 columns fails mid-way, the transaction rolls back and
 leaves no partial results. Re-running the job produces a clean state.

 Integration: How Features Stay Connected

 CSV Analysis (this feature)

 - AnalyseCsvJob streams imports/{task_uuid}/filename.csv once, stores results in csv_analysis_results
 - GET /api/imports/{import}/analysis reads cached rows — instant response, no re-computation

 PDF Generation (future)

 - GeneratePdfJob receives import_id, opens csv_import->file_path via Storage::readStream()
 - Streams headers + rows to a PDF renderer, writes output to reports/{task_uuid}.pdf
 - Sets result_path on task → served by existing TaskController@download unchanged

 Bulk Email (future)

 - SendBulkEmailBatch opens csv_import->file_path via Storage::readStream()
 - Streams rows, plucks the designated email column, dispatches Bus::batch() of email jobs
 - Same batch pattern as existing ConvertFileJob

 The file_path on CsvImport is the shared integration point. No feature needs to know how the import was processed — they all
 just stream from the same file.
