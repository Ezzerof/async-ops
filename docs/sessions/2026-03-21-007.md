 handle(Mailer $mailer): void
 1. Return early if $this->batch()->cancelled().
 2. Refresh task from DB — return early if deleted.
 3. Build BulkEmailMailable from payload (subject, body, attachment_path).
 4. $mailer->to($this->recipient)->send($mailable).

 failed(Throwable $e): void
 1. Refresh task — return early if deleted.
 2. Guard: skip if status already Failed.
 3. Append $recipient => 'failed' to payload['delivery_status'].
 4. Set status = Failed, error_message = RuntimeException message verbatim, else generic string.

● Without ->allowFailures(): When a job fails, the batch is marked as cancelled. Jobs that already ran are done (those
  recipients already received the email). Jobs queued but not yet started are skipped — those recipients get nothing. So
  it's timing-dependent and unpredictable.
                                                                                                                          
  With ->allowFailures() (our plan): Each job runs fully independently. A failure for recipient A has zero effect on      
  recipients B and C. All three jobs run to completion or exhaust their own retries. Only the specific recipient whose job
   failed doesn't receive the email.                                                                                      
                                                                                                                        
  That's exactly why ->allowFailures() is in the plan — it makes each recipient's delivery independent and the outcome    
  deterministic.
```

 Step-by-step orchestration:

 1. createTask() with type = TaskType::BulkEmail->value, initial payload:
 {subject, body, recipients, attachment_path: null}
 2. Enter try block.
 3. If $attachment present:
   - Path: 'emails/' . $task->uuid . '/attachment_' . Str::uuid() . '.pdf'
   - $attachment->storeAs(...) on local disk.
   - Update payload['attachment_path'].
 4. Build jobs: array_map(fn($email) => new SendBulkEmailJob($task, $email), $recipients)
 5. Inline Bus::batch($jobs) call (do NOT use dispatchBatchForTask() — that method assumes output files to zip, which
 doesn't apply here):

 5. Call ->allowFailures() on the batch. This is critical: without it, a single failed job cancels then() entirely, and
 the delivery report is never generated. With allowFailures(), all jobs run to completion (or exhaust retries), then()
 always fires, and per-recipient outcomes are tracked in delivery_status.

 5. Closures must capture only $taskId (int), not $task (Eloquent model) — batch closures are serialized and a stale
 model would be used. Load fresh inside each callback via Task::find($taskId).

 5. then() callback:
   - $task = Task::find($taskId) — return if null (deleted).
   - Refresh payload['delivery_status']; if null/empty, treat as all-failed (no keys with sent).
   - Generate delivery report CSV: two columns recipient,status. Write each delivery_status entry as a row.
   - Store CSV at emails/{task_uuid}/report.csv on local disk.
   - Delete the attachment file using the stored payload['attachment_path'] (not a wildcard — delete the exact path).
   - $task->update(['status' => Completed, 'progress' => 100, 'result_path' => $reportPath]).

 catch() callback (fires only on infrastructure failure — e.g. Redis crash — NOT on individual job failures when
 allowFailures() is active):
   - $task = Task::find($taskId) — return if null.
   - Guard: skip if status already Completed or Failed.
   - $task->update(['status' => Failed, 'error_message' => 'Batch processing failed unexpectedly.']).

 After ->dispatch(), store batch_id in payload.
 6. Catch block (compensating logic):
 } catch (\Throwable $e) {
     Storage::disk('local')->deleteDirectory('emails/' . $task->uuid);
     $task->delete();
     throw $e;
 }
 7. Return $task.

  Concurrent payload writes from multiple jobs
  Each SendBulkEmailJob does a refresh → merge → save on payload['delivery_status']. If two jobs for the same task finish 
  at nearly the same time, both read the same stale payload before either has written back. One write will silently       
  overwrite the other — a classic read-modify-write race. The result: one recipient's status is lost from the report.     
                                                                                                                          
  This is the same pattern used for output_files in ConvertFileJob, which has the same race. It's acceptable at small     
  scale (≤3 recipients) but worth documenting. A proper fix would use a JSON merge at the DB level (JSON_SET) or a
  separate delivery_statuses table.                                                                                       
                                                            
  then() callback reading stale delivery_status
  The then() callback loads the task fresh from DB before writing the report. But if any job's DB write (from step 5 of
  handle()) hasn't been committed yet when the callback fires, those rows will be missing from the CSV. In practice this  
  is unlikely with a synchronous DB, but possible under high load.
                                                                                                                          
  Task deleted between job completion and then() firing                                                                   
  The plan handles Task::find($taskId) === null. However if a task is deleted while jobs are mid-flight, the attachment
  file under emails/{task_uuid}/ is never cleaned up — the then() callback returns early before it reaches the deletion   
  step. The compensating catch in createBulkEmailTask() only covers the dispatch path, not post-dispatch deletion.
                                                                                                                          
  ---                                                       
  Data & logic edge cases
                                                                                                                          
  All jobs fail — then() still fires
  With allowFailures(), then() runs even if every single job failed. delivery_status will have all entries as failed, and 
  the task will be marked Completed. That's technically correct per the plan, but a task with zero successful deliveries  
  being Completed could confuse API consumers. Consider whether Completed with all-failed rows should instead be Failed.  
                                                                                                                          
  Empty recipients array after validation                                                                                 
  The request validates min:1, but the service takes an array $recipients with no independent guard. If called directly
  (not via HTTP), an empty array produces a batch with zero jobs. A zero-job batch fires then() immediately — the CSV     
  would have only a header row, and the task would be marked Completed with progress = 100. Defensive count($recipients) 
  === 0 check at the top of the method would prevent this.                                                                
                                                            
  Null delivery_status handling in then()                                                                                 
  The plan says treat null as all-failed and write unknown status rows. But if delivery_status is null, you don't know
  which recipients were intended — the original recipients array is in the payload. The CSV should be built from          
  recipients (with unknown status) rather than from delivery_status keys, otherwise the report has zero data rows instead
  of N rows showing unknown.                                                                                              
                                                            
  Attachment stored but batch dispatch throws                                                                             
  Between step 3 (attachment stored) and step 5 (batch dispatched), a failure triggers the catch block which calls
  Storage::deleteDirectory('emails/' . $task->uuid). That correctly removes the attachment. But if                        
  Bus::batch()->dispatch() itself throws after the batch is partially registered in Redis, you may have a dangling batch
  with no task record pointing to it. Unavoidable without a distributed transaction, but worth noting.                    
                                                            
  batch_id written after dispatch vs. job start timing                                                                    
  There's a window between ->dispatch() returning and the batch_id being written back to the payload. If a job starts and
  calls $this->batch()->cancelled() before the payload update completes, withLiveProgress() would see no batch_id and     
  return the task as Pending even though it's processing. Unlikely with a real queue but possible with the sync driver in
  tests.
