## Decision 3 ŌĆö `CsvImport` test failures root-caused: tests passing integer ID, model binding expecting UUID

**Context:**
Running the full test suite after the README commit revealed 21 failing tests across
`CsvImportShowTest`, `CsvImportDestroyTest`, and `CsvImportAnalyseTest`. All failures were
404s where 403s were expected.

**Challenge:**
You asked for the root cause to be identified before any fix was proposed. The failures
predated the README work and were unrelated to it.

**Outcome:**
Root cause: `CsvImport::resolveRouteBinding()` resolves the `{import}` route parameter via
the related Task's UUID (`WHERE tasks.uuid = $value`), consistent with the documented
UUID-based API design. The tests were passing `$import->id` (an integer primary key) in the
URL ŌĆö which never matches a UUID lookup, so Laravel returns 404 before the policy gate is
ever reached. Two options were presented: fix the tests (correct) or remove the custom
route binding (breaks the documented API). Option 1 was chosen ŌĆö the binding is intentional
and correct per the architecture. The three test files were updated to use `$task->uuid`
in all URL construction. All 26 failing tests passed after the fix; full suite 350/350.

## Decision 4 ŌĆö Application logging added to all jobs, `TaskService`, and `AuthController`

**Context:**
The entire codebase had zero logging calls. `config/logging.php` is standard Laravel with
a single file channel ready to use, but `Log::` was never called anywhere. On job failure,
`error_message` is stored in the DB but the full stack trace is permanently lost. There is
no observability into what jobs are doing while they run.

**Challenge:**
You identified logging as a missing feature and asked whether full application logging was
rational for the project. You wanted all job events covered. The question was where to draw
the scope boundary.

**Outcome:**
Logging was scoped to three areas: all 6 jobs (4 events each), `TaskService` (task creation
and every dispatch site), and `AuthController` (auth lifecycle). Controllers were explicitly
excluded ŌĆö they are thin HTTP adapters with no business logic worth logging. File-level I/O
was excluded as too noisy for a demo. No config changes were needed; the default single
channel writing to `storage/logs/laravel.log` is sufficient.

## Decision 5 ŌĆö Structured log format: `[Prefix] Verb phrase.` with a context array

**Context:**
With logging being added across 8 files simultaneously, a consistent message format was
needed to make the log file scannable and filterable.

**Challenge:**
The format needed to distinguish which component emitted a log line and include enough
context to be actionable without being verbose.

**Outcome:**
The format `[Prefix] Verb phrase.` with a context array was adopted throughout. Prefixes
are fixed per component: `[Report]`, `[FileConversion]`, `[DataAnalysis]`, `[CsvImport]`,
`[InvoiceGeneration]`, `[BulkEmail]`, `[TaskService]`, `[Auth]`. Every context array
includes `task_uuid` (or `$taskId` in batch callbacks where only the integer is safe to
capture). Additional context is job-specific: `user_id` for reports, `source_path` and
`target_format` for conversions, `recipient` for email jobs, `import_id` and `row_count`
for imports.

## Decision 6 ŌĆö Three log levels with explicit semantics: `info`, `warning`, `error`

**Context:**
Using a single log level throughout would make it impossible to filter signal from noise.
The distinction between an expected business outcome (bad CSV input) and an unexpected
infrastructure failure needed to be encoded in the level.

**Challenge:**
The `GenerateInvoiceJob` has an inline `catch (\InvalidArgumentException $e)` for invalid
CSV data. This is an expected failure mode ŌĆö not an infrastructure fault. Logging it at
`error` would create false alerts.

**Outcome:**
Three levels with clear semantics:
- `Log::info` ŌĆö normal lifecycle events: started, completed, dispatched.
- `Log::warning` ŌĆö expected non-fatal events: idempotency skips (task deleted, status not
  pending, batch cancelled), login failures, the `GenerateInvoiceJob` inline CSV validation
  catch. These are conditions the system handles gracefully.
- `Log::error` ŌĆö unexpected failures in `failed()` methods, always includes the full stack
  trace via `$e->getTraceAsString()`. Also used for batch callbacks that reach a
  structurally wrong state (e.g. batch completed with no output files).

## Decision 7 ŌĆö Batch callbacks and compensating logic added to logging scope after review

**Context:**
The first version of the logging plan covered job `handle()` and `failed()` methods, and
`TaskService` dispatch calls. A plan review was requested before implementation.

**Challenge:**
You asked to identify any vital missing logs. The review found that the `then()` and
`catch()` batch callbacks in `dispatchBatchForTask()` and `createBulkEmailTask()` were
completely unlogged ŌĆö there was total silence between "Job completed" (in `ConvertFileJob`)
and the task appearing as Completed in the DB.

**Outcome:**
Three additional log areas were added to the plan: (1) batch `then()` and `catch()`
callbacks for both `FileConversion` and `BulkEmail` ŌĆö covering completion, infrastructure
failure, task-deleted guards, and the empty-output edge case; (2) compensating logic in
all 6 `TaskService` `catch (\Throwable $e)` blocks that silently delete files and task
records on setup failure; (3) `deleteImport()` in `TaskService` ŌĆö user-triggered deletion.
`AuthController::register()` was also added for completeness. The plan was revised before
any code was written, keeping the implementation-from-correct-plan workflow intact.

## Decision 8 ŌĆö Three-step implementation order: jobs ŌåÆ `TaskService` ŌåÆ `AuthController`

**Context:**
The logging changes touched 8 files with ~40 total log calls. Implementing everything at
once with no structure would make review and debugging harder.

**Challenge:**
You asked for the plan to be divided into 3 steps before implementation began.

**Outcome:**
Three steps in ascending complexity:
- **Step 1 ŌĆö Jobs**: All 6 job files. Consistent 6-call pattern (7 for `GenerateInvoiceJob`).
  Largest number of files but lowest per-file complexity ŌĆö each job follows the same
  structure.
- **Step 2 ŌĆö `TaskService`**: Single file, ~22 calls. Most complex: covers task creation,
  8 dispatch sites, 2 sets of batch callbacks, 6 compensating catch blocks, and
  `deleteImport()`. Implemented and verified before moving on.
- **Step 3 ŌĆö `AuthController`**: Smallest change ŌĆö 4 calls across `register()`, `login()`,
  and `logout()`. Isolated and independent of the other two steps.

Each step was shown with a description and confirmed before implementation, keeping the
analysis-then-implement workflow consistent with the project convention.
