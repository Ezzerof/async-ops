## Decision 1 ‚Äî `TaskService` split into `createTask()` + `dispatchBatchForTask()`

**Context:**
The original `createAndDispatchBatch()` method created a task and dispatched the batch in one
call. Building `ConvertFileJob` instances requires both the task (for its UUID) and the stored
file paths ‚Äî but files must be stored under `uploads/{task_uuid}/` before jobs can be built,
which requires the task to exist first.

**Challenge:**
The original design created a chicken-and-egg dependency: `ConvertFileJob` needed a `Task`
instance, but the controller had to store files under the task's UUID before it could build
the jobs. This forced a structural decision about where the split in responsibility should live.

**Outcome:**
`createAndDispatchBatch()` was refactored into two discrete methods: `createTask()` ‚Äî a thin
helper that persists the task record without dispatching anything ‚Äî and `dispatchBatchForTask()`
‚Äî which owns the batch wiring (`then`/`catch` callbacks and `batch_id` update). The original
`createAndDispatchBatch()` now delegates to both, preserving the existing report generation
path without behaviour change. The controller creates the task first, stores files, then
dispatches ‚Äî each step in a defined order.

## Decision 2 ‚Äî No `DB::transaction`; compensating logic in `TaskService` instead

**Context:**
After identifying that a failed `$file->store()` call could leave an orphaned `Task` record
in `Pending` state with no batch ever dispatched, the question arose whether wrapping the
entire store-and-dispatch flow in `DB::transaction` was the right safety net.

**Challenge:**
You asked directly: can we wrap store + dispatch in a `DB::transaction`? The concern was both
correctness and whether doing so would violate SRP.

**Outcome:**
`DB::transaction` was rejected. It only protects database consistency ‚Äî it cannot roll back
filesystem writes (`$file->store()` already persisted files to disk) or queue dispatches
(Redis is unaffected by a DB rollback). Using it would give a false sense of safety while
leaving files orphaned on disk. The correct pattern for operations that span multiple
infrastructure boundaries (DB + filesystem + queue) is compensating logic: explicitly undo
previous steps on failure. `TaskService::createConversionTask()` wraps the entire flow in a
`try/catch` ‚Äî on any failure it deletes stored upload files and the task record before
re-throwing. The controller remains blind to recovery details; it only handles the HTTP
response.

## Decision 3 ‚Äî `TaskType` backed string enum introduced

**Context:**
`ConversionController` hardcoded `'file_conversion'` as a plain string when creating tasks.
The same string was duplicated across the controller, tests, and the factory. If
`TaskController` or any other layer needed to branch on task type, the string would spread
further.

**Challenge:**
The issue was flagged during the `ConversionController` SOLID analysis as a magic string
violation (OCP). The question was how to standardise it without a migration.

**Outcome:**
`TaskType` backed string enum was introduced alongside the existing `TaskStatus` enum, with
cases `UserExport = 'user_export'` and `FileConversion = 'file_conversion'`. All magic string
occurrences were replaced: `StoreTaskRequest` switched from `Rule::in(['user_export'])` to
`Rule::enum(TaskType::class)`, the factory uses `TaskType::UserExport->value`, and tests
reference `TaskType::FileConversion->value`. The enum is the single source of truth ‚Äî adding
a new task type extends validation automatically.

## Decision 4 ‚Äî Job building and file storage moved from controller to `TaskService`

**Context:**
The initial `ConversionController::store()` implementation instantiated `ConvertFileJob`
directly and called `$file->store()` inline. A senior-developer SOLID analysis identified
both as responsibility violations: the controller knew the job's constructor signature (tight
coupling, SRP + DIP violation) and owned infrastructure decisions about disk name and path
layout (SRP violation).

**Challenge:**
You asked for the analysis before any fixes, then confirmed all five issues should be
addressed. The scope question was whether file storage warranted its own `FileUploadService`
or whether `TaskService` was sufficient.

**Outcome:**
Both concerns were absorbed into `TaskService`. `dispatchConversionBatch()` owns job
construction (the controller passes domain data ‚Äî task, paths, format ‚Äî and never imports
`ConvertFileJob`). `storeUploadedFiles()` owns the disk name and path layout. The resulting
controller `store()` method is three lines: parse the enum, call
`taskService->createConversionTask()`, return a 201 response. No import of jobs, storage
facades, or task types.

## Decision 5 ‚Äî `withLiveProgress()` moves batch logic to `TaskService`; returns a clone, never mutates route-bound model

**Context:**
After step 11 (batch progress derivation), `TaskController::show()` contained a private
`hydrateBatchProgress()` method that called `Bus::findBatch()` directly and set `$task->status`
and `$task->progress` on the Eloquent model. A SOLID analysis identified two problems: the
controller had acquired orchestration knowledge it should not have (SRP), and the method was
mutating the route-bound model instance that Laravel resolves and shares across the request
lifecycle (correctness bug).

**Challenge:**
You asked for fix 1 (SRP ‚Äî move `hydrateBatchProgress`) and fix 6 (mutation ‚Äî route-bound
model) to be applied. Both had the same root cause and a single refactor addressed both.

**Outcome:**
`TaskService::withLiveProgress(Task): Task` was introduced. It looks up the batch, and if
the batch is in-flight, returns a `clone` of the task with `status` and `progress` injected ‚Äî
the original instance is never touched. If there is no batch ID, the batch cannot be found,
or the batch has finished, the original task is returned unchanged. The controller `show()`
became a two-liner. The `Bus` facade import and `TaskStatus` import were removed from the
controller entirely. `clone $task` before mutating for response is now an explicit
architectural rule for all service methods that need to enrich a model for output.

## Decision 6 ‚Äî MIME resolution and download filename logic moved to `TaskService::resolveDownloadMeta()`

**Context:**
`TaskController::download()` had `text/csv` and `report-{uuid}.csv` hardcoded. Adding
conversion task support required branching on task type inside the download method, which
the SOLID analysis flagged as an OCP violation ‚Äî the method was not open for extension without
modification.

**Challenge:**
Step 12 had to address the concrete bug (wrong Content-Type and filename served for conversion
tasks) without making the controller aware of task types or MIME mappings.

**Outcome:**
`TaskService::resolveDownloadMeta(Task): array` was added, returning `['filename' => ...,
'content_type' => ...]`. It owns a `MIME_MAP` private const mapping all supported extensions
to their correct Content-Type headers. For `user_export` tasks it returns the existing
`report-{uuid}.csv` / `text/csv`. For `file_conversion` tasks it derives the extension from
`result_path`, maps it through `MIME_MAP`, and falls back to `application/octet-stream` for
unknown extensions. The controller calls one service method and remains blind to task types
and MIME logic.

## Decision 7 ‚Äî Session retrospective: workflow patterns identified

**Context:**
At the end of the session you asked for an honest analysis of the collaboration and what
could be improved in future sessions. You noted you were new to Claude and unaware of all
available capabilities.

**Challenge:**
The session had surfaced several reactive fixes: the chicken-and-egg problem mid-implementation,
the orphaned-task bug identified after the controller was written, and the route-bound mutation
bug found only during the SOLID review.

**Outcome:**
Ten workflow improvements were identified. The highest-priority ones that directly address the
reactive issues in this session:

1. **CLAUDE.md** ‚Äî loading project context, Docker commands, conventions, and architectural
   rules at session start eliminates cold re-explanation. The file was planned at the end of
   this session.
2. **Plan Mode before non-trivial implementation** ‚Äî the chicken-and-egg conflict in step 8
   would have been caught in a research-then-design phase rather than discovered mid-code.
3. **Analysis ‚Üí discuss approach ‚Üí implement** ‚Äî the `DB::transaction` discussion before
   fixing was the correct pattern; making it a consistent habit avoids Claude assuming an
   approach you disagree with.
4. **Role-framing for deep analysis** ‚Äî "Analyse as a Senior Developer" and "You are a Senior
   Test Engineer" both produced significantly more thorough, prioritised output than generic
   review requests.
5. **Ask Claude to review its own output** ‚Äî systematically asking "what tests are missing for
   what you just implemented?" after each step catches gaps before they accumulate.
