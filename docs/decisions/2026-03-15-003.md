# Session 3 — File Conversion Feature Design

**Date:** 2026-03-15
**Branch:** `feature/file-conversion`
**Transcript:** [2026-03-15-003](../sessions/2026-03-15-003.md)

---

## Decision 1 — Always `Bus::batch()` regardless of file count

**Context:**
Three dispatch strategies were evaluated for the conversion feature: always use `Bus::batch()`
even for a single file, branch on file count (direct dispatch vs batch), or split files
across chunked batches at a threshold. The question was which approach best balanced
simplicity and correctness for a 1–10 file cap.

**Challenge:**
You asked for a thorough comparison of all three options, including trade-offs on simplicity
and scalability, before committing to an approach.

**Outcome:**
Always `Bus::batch()` was chosen. The uniform code path is the decisive argument: conditional
dispatch (Option B) creates two parallel code paths through dispatch, progress derivation in
`show()`, and test coverage. With a hard 10-file cap the overhead of a single `job_batches`
record per request is negligible. The free `$batch->progress()` and clean `then`/`catch`
callbacks are additional gains. Chunked batches (Option C) were rejected as overkill at this
scale.

---

## Decision 2 — Progress derived from `Bus::findBatch()`, not per-job DB writes

**Context:**
Three approaches to keeping the `progress` integer up to date while batch jobs run were
compared: query the batch API in `show()`, have each job increment `task->progress` on
completion, or leave progress at 0 until the `then` callback sets it to 100.

**Challenge:**
You asked for analysis before approving. The concurrency hazard of per-job DB writes was the
central concern — two jobs completing simultaneously can clobber each other's progress value
on the same row without an explicit lock.

**Outcome:**
`Bus::findBatch($batchId)->progress()` is called in `TaskController::show()` when
`type === 'file_conversion'` and `batch_id` is present in the payload. No job-side DB writes.
The batch API was designed exactly for this pattern and eliminates the hot-row race condition.
Edge case: if the batch has been pruned by `queue:prune-batches` and the task is `Completed`,
100 is returned — batch absence on a completed task is harmless. Binary 0→100 (Option C) was
rejected because it defeats the purpose of the progress field.

## Decision 3 — Single `target_format` per request

**Context:**
Two shapes were evaluated for the conversion request: a single `target_format` applied to
all uploaded files, or a nested multipart structure where each file carries its own desired
output format.

**Challenge:**
You asked for a comparison before committing. The concern was whether the single-format
constraint would be a genuine limitation or a reasonable design boundary for a demo.

**Outcome:**
Single `target_format` per request. PHP and Laravel do not handle nested multipart cleanly,
and per-file format mapping would add significant validation complexity, job constructor
pairing, and payload schema changes for a constraint that is a UX preference rather than a
technical requirement. The established pattern for mixed conversions is separate requests,
not a complex nested body.

---

## Decision 4 — Conditional result delivery: direct file for one upload, zip for multiple

**Context:**
Three download strategies were evaluated: conditional (single file streamed directly, multiple
zipped), always zip regardless of file count, and no zip at all (client makes N separate
download requests).

**Challenge:**
You asked for a comparison before approving. The trade-off between code simplicity (always
zip has fewer branches) and UX quality (forcing a client to unzip a single converted file)
was the main axis.

**Outcome:**
Conditional delivery was chosen. The branch is small and fully contained in the `then`
callback and `download()` response headers. Forcing clients to unzip a single file (Option B)
is unnecessary friction. Option C breaks multi-file ergonomics entirely by requiring multiple
client requests for a batch result. The `result_path` column points to the converted file
directly for single uploads, and to `result.zip` for multiple.

## Decision 5 — All-or-nothing failure with per-job retries

**Context:**
Three failure strategies were evaluated: all-or-nothing (any job failure marks the whole task
`Failed`), partial success (complete with whatever succeeded, report failures in payload), and
all-or-nothing with per-job retries before escalating to batch failure.

**Challenge:**
You asked for a comparison before committing. The question was whether partial success was
worth the added state machine complexity.

**Outcome:**
All-or-nothing + per-job retries (Option A + C). All-or-nothing keeps the `TaskStatus` state
machine clean and binary — a `Completed` task with partial output is semantically misleading,
doubles test coverage requirements, and makes `show()` and `download()` harder to reason
about. Per-job `$tries = 3` with exponential backoff mirrors the existing `GenerateReportJob`
pattern and handles transient I/O errors before they ever reach the batch `catch` callback.
Only truly unrecoverable failures escalate.

## Decision 6 — Dedicated `ConversionController` for upload; polling and download stay in `TaskController`

**Context:**
Three controller structures were evaluated: a dedicated `ConversionController` for the upload
entry point, adding a `conversion()` method to the existing `TaskController`, or extending
the unified `POST /api/tasks` endpoint to accept file uploads.

**Challenge:**
You asked for a comparison. The concern was whether a dedicated controller was over-engineering
or the correct separation of concerns.

**Outcome:**
Dedicated `ConversionController` for `POST /api/conversions`. `multipart/form-data` and
`application/json` are fundamentally different content types — a unified endpoint would
require content-type detection logic and make `StoreTaskRequest` deeply conditional. The
existing `TaskController::show()` and `TaskController::download()` are unchanged (except for
the progress branch in `show()`) because they operate on the Task resource, not on the
conversion action. Adding conversion handling to `TaskController` (Option B) would make the
controller a catch-all that grows with every new task type.

## Decision 7 — `ConversionFormat` backed string enum for target; source format inferred, not enumerated

**Context:**
You asked for a motivated senior recommendation on whether to introduce an enum for
`target_format`. The question was whether an enum was genuine value or over-engineering for
this use case. The handling of source format (the format of the uploaded file) was considered
as a separate question.

**Challenge:**
You explicitly requested justification rather than a simple yes/no, and asked specifically
whether source formats should also be enumerated.

**Outcome:**
Backed string enum `ConversionFormat` with cases `Json = 'json'`, `Csv = 'csv'`,
`Xml = 'xml'`, `Simulated = 'simulated'`. The enum earns its keep on three grounds:
(1) `Rule::enum(ConversionFormat::class)` in `StoreConversionRequest` is declarative — the
enum is the single source of truth, so adding a case automatically extends validation without
updating a hardcoded array; (2) `outputExtension(): string` and
`isRealConversion(string $sourceExtension): bool` methods centralise routing logic on the
enum rather than scattering conditionals across the service; (3) `FileConversionService` takes
a typed `ConversionFormat` argument — once `StoreConversionRequest` casts the input,
downstream code is guaranteed a known value.

Source format is deliberately NOT enumerated. It is inferred from `pathinfo()` on the
uploaded filename, not provided by the user, so there is nothing to validate against an enum.
An enum here would require updating both the enum and the `mimes` validation rule whenever a
new fake format is added, whereas only the `mimes` rule needs updating without one. The
project convention of no magic strings applies to user-supplied values; file extensions inferred
at runtime are not in that category.
