## Decision 1 — Single job pattern (`GenerateReportJob`), not `Bus::batch()`

**Context:**
AsyncOps already had two async task patterns: the single-job `GenerateReportJob` and the
batched `ConvertFileJob` via `Bus::batch()`. The data analysis feature processes a single
uploaded CSV file and needed to choose which pattern to follow.

**Challenge:**
The architecture request specified Redis queues, progress tracking, and async processing. The
question was whether the file-by-file parallelism of `Bus::batch()` was warranted here.

**Outcome:**
The `GenerateReportJob` single-job pattern was chosen. Data analysis is inherently a single
sequential computation — there is no parallel work to distribute across multiple jobs. The
batch pattern exists to fan out independent jobs and track completion across them; for a
single file with a streaming algorithm, it would add `job_batches` overhead and `batch_id`
payload handling with no benefit. Progress is written to the DB at coarse milestones (0, 50,
90, 100) inside `AnalyseDataJob`, identical to `GenerateReportJob`. No new migrations needed.

## Decision 2 — Phased implementation: foundation before job and service

**Context:**
The implementation plan covered seven files across four layers: enum, FormRequest, controller,
route, service, job, and tests. The question was in what order to build them.

**Challenge:**
You explicitly directed a layered build — implement only the foundation (enum, request,
controller, route) in the first step, then add the TaskService methods, then the pure service,
then the job, then tests.

**Outcome:**
The layered order was adopted: enum → FormRequest → controller → route → TaskService methods
→ `DataAnalysisService` → `AnalyseDataJob` → tests. The rationale is that the foundation
establishes the HTTP contract first — URL, input shape, and enum value. Auth, validation, and
routing are fully functional and testable before any async logic is written. Building the job
and service on top of a defined contract is cleaner than wiring everything at once. This also
mirrors the build order used for the file conversion feature.

## Decision 3 — `DataAnalysisService` as a pure computation class

**Context:**
The service could have been designed with access to `Storage`, the DB, or the task model —
similar to how `TaskService` orchestrates all three. The alternative was a class with no
framework dependencies that receives an absolute file path and returns a plain PHP array.

**Challenge:**
You required the service to be "pure computation" with no database access, no Storage facade
calls, and no queue logic. The constraint was explicit.

**Outcome:**
`DataAnalysisService::analyse(string $csvPath): array` receives an absolute filesystem path
and returns the statistics array. All Laravel concerns are resolved before the call:
`AnalyseDataJob` converts the relative stored path to an absolute path via
`Storage::disk('local')->path(...)` before passing it in. The service opens the file with
`fopen()` directly, reads it with `fgetcsv()`, and returns plain PHP arrays. No `Storage`
facade, no Eloquent, no queue interaction. The `finally` block guarantees `fclose()` always
runs, preventing file handle leaks on exception. This makes the service fully unit-testable
with real filesystem fixtures, without any Laravel bootstrapping.

## Decision 4 — Result stored as a file at `result_path`, not in the `payload` column

**Context:**
The analysis output is a JSON structure with per-column statistics. For string columns,
`value_counts` is an unbounded frequency map — a column of UUIDs would produce one entry per
row. Two storage options existed: write the stats into `task->payload` as a JSON column, or
write them to disk as `analyses/{task_uuid}/result.json` and set `result_path`.

**Challenge:**
The architecture question was whether the `payload` column was the right place for potentially
large computed output.

**Outcome:**
Results are written to disk as `analyses/{task_uuid}/result.json`. The existing
`TaskController::download()` and `resolveDownloadMeta()` serve the file via the same path
already used for CSV reports and conversion outputs — no changes to the download endpoint.
`resolveDownloadMeta()` received a third branch: `TaskType::DataAnalysis` returns
`analysis-{uuid}.json` and `application/json`. Keeping large, variable-size output out of the
`payload` column keeps the DB row small regardless of CSV cardinality, and is consistent with
the project's pattern of storing all file outputs at `result_path`.

## Decision 5 — `fgetcsv` streaming with single-pass accumulation

**Context:**
Three approaches exist for reading a CSV in PHP: load the whole file into memory with
`file()` or `str_getcsv`, use `SplFileObject`, or stream row by row with `fgetcsv` in a
`while` loop. The choice affects memory usage for large files.

**Challenge:**
The service requirement was "pure computation" with no framework dependencies. The constraint
was also that it should handle arbitrarily large files without silently hitting `memory_limit`.

**Outcome:**
`fgetcsv()` streaming was chosen. One row is in memory at a time — the full CSV is never
loaded as an array. A single pass accumulates all column statistics: `null_count`, an
`all_numeric` boolean flag (flips false on the first non-numeric value and never reverts,
allowing `is_numeric()` to be skipped for subsequent values in disqualified columns), running
`min`, `max`, `sum`, `numeric_count`, and a `value_counts` frequency map. `finalise()` runs
once after the loop to convert raw accumulators into the output shape. `average` is computed
then rather than as a running mean, avoiding the need for additional state during streaming.

## Decision 6 — `mimes:csv` validation (not content-based `mimetypes`)

**Context:**
Two Laravel validation rules exist for file type checks: `mimes:csv` (checks the file
extension against a known MIME type mapping) and `mimetypes:text/csv,text/plain` (inspects
the file's actual content via PHP's `finfo`). A file named `malicious.csv` containing
non-CSV binary content would pass `mimes:csv`.

**Challenge:**
The structured review of the implementation flagged this as an informational finding —
consistent with the project standard but not content-based.

**Outcome:**
`mimes:csv` was retained. `StoreConversionRequest` uses the same `mimes` approach for all its
formats, making this consistent with the project standard. For a technical assessment with
controlled inputs, extension-based validation is defensible. The finding was explicitly
documented in the review rather than silently accepted — the decision threshold is: use
`mimetypes` when the application has a meaningful attack surface from user-provided files
processed server-side; use `mimes` for demos and assessments where consistency matters more.

## Decision 7 — 90% progress milestone caught in plan-vs-implementation review

**Context:**
The original architecture plan specified four progress milestones in `AnalyseDataJob`:
0% (Processing set), 50% (after `analyse()` returns), 90% (after `Storage::put()`), 100%
(Completed set). When the job was implemented, it jumped directly from 50% to 100% — the
90% write after the file was persisted was omitted.

**Challenge:**
You requested a structured review of all data analysis changes against the original plan,
covering alignment, layer design, and security. The missing milestone was surfaced during the
plan alignment check before tests were written.

**Outcome:**
The 90% milestone was added before tests were implemented. The fix was intentionally
requested ahead of tests because the test plan listed four milestone assertions — writing
tests against the incomplete implementation would have required rewriting them after the fix.
The review-before-tests sequence prevented that rework. The structured review format
(plan alignment → layer design → validation/security) is now an established pattern: it
catches implementation drift at the cheapest possible point, before test coverage locks in
the wrong behaviour.

## Decision 8 — Type detection: `is_numeric()` with all-empty column fallback to string

**Context:**
Column types need to be determined from the data. Candidates included: PHP's `is_numeric()`
(accepts integers, floats, scientific notation), `ctype_digit()` (integers only), regex
patterns, or a multi-type detection pass.

**Challenge:**
The requirement was to support two types — numeric and string — with a clear, defensible rule.
The edge case of an entirely empty column (every cell blank) required an explicit decision.

**Outcome:**
A column is classified as numeric only if `all_numeric` is still `true` after the loop (no
non-numeric value was ever seen) AND `numeric_count > 0` (at least one non-empty value
existed). The second condition is the critical guard: an all-empty column never triggers
`is_numeric()`, so `all_numeric` remains `true` by default — without the `numeric_count > 0`
check, an empty column would be classified as numeric with no values, causing a division-by-
zero when computing average. All-empty columns fall through to string, which is a safe
descriptive default. The assumptions are documented: `is_numeric()` accepts scientific
notation and leading zeros; `value_counts` is unbounded for string columns; dates and
booleans fall back to string. These are explicit trade-offs acceptable for the demo scope.
