# Session 6 — CSV Import Feature Architecture

**Date:** 2026-03-18
**Branch:** `feature/csv-import`
**Transcript:** [2026-03-18-006](../sessions/2026-03-18-006.md)

---

## Decision 1 — Hybrid storage: file on disk + cached analysis results (not a `csv_rows` table)

**Context:**
The initial architecture stored every parsed CSV row as a JSON object in a `csv_rows` table
(`data` column: `{"name":"Alice","amount":"120.50"}`). A critical self-review of that design
was requested before any implementation began.

**Challenge:**
You asked to critically review the architecture and challenge the design. Five weaknesses were
identified: (1) JSON sub-fields cannot be indexed without a generated column, making
aggregation queries a full scan; (2) no transaction around chunk inserts leaves orphaned rows
on failure; (3) no delete endpoint means the table grows forever with no pruning path;
(4) key names are repeated in every row, inflating storage; (5) if downstream features only
need to iterate rows in order, a `csv_rows` table is unnecessary — a file on disk is
sufficient.

**Outcome:**
The `csv_rows` table was dropped entirely. The revised architecture uses a hybrid approach:
the validated CSV is stored permanently on disk at `imports/{task_uuid}/filename.csv`. Column
statistics are computed once by `AnalyseCsvJob` and cached in a dedicated
`csv_analysis_results` table. Downstream features (PDF generation, bulk email) stream
directly from `file_path` on the `CsvImport` record — no feature needs to know how the
import was processed. This eliminates unbounded table growth, JSON query performance issues,
and the orphaned-row failure mode in a single structural change.

## Decision 2 — `csv_analysis_results` table with typed `decimal` columns, not a JSON blob

**Context:**
Once the `csv_rows` table was dropped, a place for cached analysis output was still needed.
The options were: a JSON column on `csv_imports`, a single `results` JSON file on disk, or a
dedicated normalised table with one row per column per import.

**Challenge:**
The goal was a structure that is queryable, typed, and fast to read. JSON blobs on the import
record would conflate import metadata with analysis output, and a file on disk would require
parsing on every read.

**Outcome:**
A dedicated `csv_analysis_results` table was introduced with typed `decimal(15,4)` columns
for `min`, `max`, `sum`, and `average` (all nullable, to handle columns that have no numeric
values), an unsigned integer `count`, and a string `column_name`. A unique index on
`(csv_import_id, column_name)` enforces one result row per column per import and enables
efficient re-analysis by deleting then re-inserting. The GET analysis endpoint reads cached
rows — instant response with no re-computation.

## Decision 3 — Two-phase validation: sync FormRequest + async job content checks

**Context:**
CSV validation has two distinct layers: format and size (can be checked immediately on the
HTTP request) and content integrity (requires parsing the file, which is too slow for a
synchronous response).

**Challenge:**
The architecture needed to decide where each class of validation error surfaces and what
signal the client receives in each case.

**Outcome:**
Phase 1 (sync, `StoreImportRequest`, returns 422): MIME type (`text/csv` or
`application/csv`), extension (`csv`), and max size (10 MB). Phase 2 (async,
`ImportCsvJob`, marks task `Failed` with error message): file still exists on disk, file is
parseable as CSV, header row present with at least one column and no duplicate names, every
data row's column count matches the header count. Malformed rows fail the entire job — this
is an atomic guarantee, consistent with the all-or-nothing failure pattern used throughout
the project. Clients poll `GET /api/tasks/{uuid}` to observe the async failure.

---

## Decision 4 — Two separate jobs: `ImportCsvJob` (validate + store) and `AnalyseCsvJob` (compute stats on demand)

**Context:**
The feature involves two distinct async operations: ingesting and validating the file, and
computing column statistics from it. The question was whether to combine them into one job or
keep them separate.

**Challenge:**
The integration goal was to make features feel connected — the user imports a CSV and can
later run analysis on it. Analysis needed to be triggerable on demand and re-runnable, which
ruled out always running it as part of import.

**Outcome:**
Two separate single jobs were chosen, each triggered by its own endpoint. `ImportCsvJob`
validates headers and rows, copies the file to its permanent location, creates the
`CsvImport` record, and marks the task complete. `AnalyseCsvJob` is triggered separately by
`POST /api/imports/{import}/analyse`, streams the validated file, computes per-column
numeric stats via `CsvAnalysisService`, and caches them in `csv_analysis_results`. Neither
job uses `Batchable` — both are sequential, single-file operations that mirror the
`GenerateReportJob` structure. Separating them allows re-analysis without re-import and keeps
each job's responsibility clear.

## Decision 5 — `ImportCsvJob`: no transaction needed; single atomic `CsvImport::create()` at the end

**Context:**
A previous session established that `DB::transaction()` is the wrong tool for operations
spanning filesystem and DB. The import job stores a file and creates a DB record — this
interaction needed an explicit atomicity decision.

**Challenge:**
The question was how to prevent orphaned DB state if the job fails partway through.

**Outcome:**
`CsvImport::create()` is called once, at the very end of the job, only after the file has
been validated, counted, and moved to its permanent path. If the job fails before that point
— during header validation, row streaming, or the file copy — there is no partial DB state
to clean up. A single `create()` at the end is inherently atomic. `DB::transaction()` would
add overhead and false safety for a write that only happens once the work is complete. The
`failed()` hook deletes the uploaded file if it still exists on disk, preventing orphaned
uploads.

---

## Decision 6 — `AnalyseCsvJob`: `DB::transaction()` wraps result inserts

**Context:**
Unlike `ImportCsvJob`, the analysis job writes multiple rows to `csv_analysis_results` — one
per column. A 10-column CSV writes 10 rows. If the insert fails mid-way, the table would
hold partial results for the import.

**Challenge:**
Re-analysis support (running analysis a second time on the same import) also required a clean
strategy for replacing existing results without leaving partial state.

**Outcome:**
`DB::transaction()` wraps the entire result insertion: delete any existing results for this
import, then `CsvAnalysisResult::insert($results)` as a bulk insert. If the insert fails,
the transaction rolls back and leaves either the old results intact or no results — never
partial. This is the correct use of `DB::transaction()`: protecting DB-only writes where all
steps are SQL operations and rollback is meaningful. This is explicitly distinct from the
previous session's rejection of `DB::transaction()` for filesystem + DB operations.

---

## Decision 7 — `DELETE /api/imports/{import}` endpoint added from the start

**Context:**
The initial architecture had no delete endpoint. The critical review identified this as a
blocker: without a delete path, `csv_imports` records and their associated files accumulate
indefinitely. Cascade deletes from `csv_imports` handle `csv_analysis_results` rows, but a
`DELETE` route must trigger the cascade.

**Challenge:**
The concern was unbounded table and disk growth. Users have no way to prune imports they no
longer need.

**Outcome:**
`DELETE /api/imports/{import}` was added to the API surface from the start of the revised
plan, not as an afterthought. The `destroy()` action deletes the `CsvImport` record
(cascading to `csv_analysis_results`), then deletes the file from disk via
`Storage::delete()`, and returns 204. Access is controlled by `CsvImportPolicy::delete()`,
which checks `user_id` ownership.

## Decision 8 — `file_path` on `CsvImport` as the shared integration point for downstream features

**Context:**
The three features that depend on imported CSV data — CSV analysis, PDF generation, and bulk
email sending — each need access to the same data. The options were: query a shared DB table,
pass data through the queue payload, or give each feature a reference to the stored file.

**Challenge:**
The stated goal was for features to "feel connected, not like unrelated demos." The
integration point needed to be stable, simple, and not require features to know about each
other's implementation.

**Outcome:**
`file_path` on the `CsvImport` model is the single integration point. Every downstream
feature receives an `import_id`, loads the `CsvImport` record, and streams the file via
`Storage::readStream()`. `AnalyseCsvJob` reads it for statistics. A future `GeneratePdfJob`
streams headers and rows into a PDF renderer. A future `SendBulkEmailBatch` plucks the email
column and dispatches one job per row using the same `Bus::batch()` pattern as
`ConvertFileJob`. No feature needs to know how the import was processed or what schema the
rows follow — they all stream from the same file.

---

## Decision 9 — Phased build order: DB foundation → import pipeline → read/delete → analysis → guards + tests

**Context:**
The full feature spans 26 steps across 10+ files. An unordered implementation would require
revisiting files multiple times and leave the system in an untestable intermediate state.

**Challenge:**
You asked for a phased roadmap with each phase independently testable before proceeding.

**Outcome:**
Five phases were defined in dependency order:
- **Phase 1** — Migrations, models, enum cases. No behaviour; confirms schema is correct.
- **Phase 2** — Upload request, `TaskService`, `ImportCsvJob`, controller, route. First end-to-end queue pipeline.
- **Phase 3** — Policy, show, destroy, routes. Makes imports accessible and prunable.
- **Phase 4** — `CsvAnalysisService`, `AnalyseCsvJob`, second `TaskService` method, controller actions, routes.
- **Phase 5** — `resolveDownloadMeta` guard for new task types, feature and unit tests, regression.

Each phase ends with a defined checkpoint (schema valid, or POST → queue → record exists, or
GET returns data). Phase 1 was implemented in this session; phases 2–5 were deferred.
