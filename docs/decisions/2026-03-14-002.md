## Decision 1 — TaskService layer added for orchestration

**Context:**
The initial plan put task creation and job dispatch directly in the controller. The question was
whether a dedicated service class was justified for a three-route feature, or whether the controller
could own the logic inline.

**Challenge:**
You asked whether the project could exist without a Service class and what the trade-offs were.
The concern was whether introducing a service layer was over-engineering for a demo.

**Outcome:**
`TaskService` was introduced with a single `createAndDispatch()` method. The deciding argument was
reusability across entry points — the same creation logic would eventually be needed from Artisan
commands or scheduled jobs, not just HTTP. Testability of the business logic in isolation was the
secondary reason. The controller was kept as a pure HTTP translator with no orchestration logic.

## Decision 2 — API Resources skipped

**Context:**
Laravel API Resources provide a dedicated transformation layer between Eloquent models and JSON
responses. Without one, controllers return raw Eloquent JSON, which exposes every column including
internal fields and changes silently if a column is renamed.

**Challenge:**
You asked whether the project could work without Resources and what the trade-offs were. The
concern was stability of the API contract and control over response shape.

**Outcome:**
Resources were deferred. For a demo with a stable schema and no external clients, the indirection
did not justify the extra files. The senior rule of thumb applied: use a Resource from day one on
any endpoint consumed by an external client; for an internal assessment demo, raw Eloquent JSON is
defensible. The decision was recorded explicitly rather than left undocumented.

## Decision 3 — Repository Pattern skipped

**Context:**
The Repository pattern sits between domain logic and Eloquent, providing a defined interface for
all DB interaction on a given model. The question was whether to introduce one alongside the new
`TaskService`.

**Challenge:**
You asked for the trade-offs of skipping it before the plan was updated. You wanted to understand
when the pattern is warranted rather than taking a blanket position.

**Outcome:**
The Repository was deferred. The honest senior verdict was that Eloquent's Model already implements
the Active Record pattern — adding a repository on top is adding a pattern on top of a pattern,
a common over-engineering trap in Laravel. Feature tests hit a real database and cover the gap
adequately. The decision threshold established: introduce a Repository when query complexity or the
need for true unit isolation (no DB, ever) makes the absence painful — not speculatively.

## Decision 4 — Redis with AOF persistence over Database Queue

**Context:**
The project already had a running Redis Docker container. The database queue driver was the Laravel
default. The question was which queue backend to use and how to configure it.

**Challenge:**
No direct pushback — you asked for a thorough explanation of the trade-offs rather than a simple
recommendation, including the durability risk of Redis without persistence configured.

**Outcome:**
Redis with AOF persistence was chosen. The concrete advantage: Redis uses a blocking pop (BLPOP)
so workers wake the instant a job arrives, versus the database queue which polls MySQL with
`SELECT FOR UPDATE` row locking under concurrent workers. AOF with `--appendfsync everysec` was
configured on the Docker container so in-flight jobs survive a crash by replaying the append-only
log on startup. The named Docker volume (`asyncops-redis-data`) ensures the AOF file persists
independently of the container lifecycle. `QUEUE_CONNECTION=sync` in `phpunit.xml` was kept
unchanged — tests never touch Redis.

## Decision 5 — Laravel Sanctum + API tokens over session-based auth

**Context:**
Task routes were initially placed in `routes/web.php` under the `auth` middleware, using Laravel's
default session guard. The question arose whether they should move to `routes/api.php` with
Sanctum token authentication.

**Challenge:**
You asked whether the project needed `api.php` at all, and then — after hearing the trade-offs —
asked what changes a Sanctum transition would require before committing.

**Outcome:**
The routes were migrated to `routes/api.php` under `auth:sanctum`. The deciding factors: stateless
clients (SPAs, mobile apps, third-party services) cannot maintain a browser session; they
authenticate with a Bearer token. The practical consequence of the old approach was also concrete —
an unauthenticated request to a `web.php` route returned a 302 redirect to a login page that does
not exist; `auth:sanctum` returns 401. A minimal `AuthController` was added for login and logout.
All domain logic — the job, service, policy, and controller — was unaffected by the swap.

## Decision 6 — UUID as route key alongside integer primary key

**Context:**
Task routes used the integer `id` as the route parameter (`/tasks/{task}`), which means sequential
IDs appeared in URLs and download filenames. You raised the concern that users could infer the
total number of tasks from the filename in the download dialog.

**Challenge:**
You proposed using a UUID in the filename. The question was how to implement it — as the primary
key or as a separate column — without reworking the completed migration, factory, and job.

**Outcome:**
Option B was chosen: a separate `uuid` column alongside the auto-increment integer `id`. The
integer remains the primary key for DB performance (foreign keys, joins, index locality). The UUID
is generated automatically in the `Task::booted()` creating event so it is never mass-assignable.
`getRouteKeyName()` returns `'uuid'` so Laravel's route model binding resolves `{task}` via
`WHERE uuid = ?` in all three routes — the integer `id` is never exposed in any URL or filename.
Download filenames become `report-{uuid}.csv`, completely opaque to the user.

## Decision 7 — php://temp stream + fputcsv for CSV generation with 10% progress thresholds

**Context:**
The original plan built the entire CSV as a PHP string variable in memory before calling
`Storage::put()`. For large user tables this would silently hit `memory_limit` and fail. Three
approaches were evaluated: in-memory string, `php://temp` stream, and a real temp file on disk.

**Challenge:**
You asked for a recommendation with clear trade-offs before approving the updated plan. The
progress tracking approach (fixed row interval vs. percentage threshold vs. Redis) was also
evaluated separately.

**Outcome:**
`php://temp` stream with `fputcsv` was chosen: memory stays flat, `fputcsv` handles CSV encoding
(commas in names, embedded quotes) correctly, and Flysystem reads the stream in chunks — the full
file is never in PHP memory at any point. No temp file cleanup is required. Progress is updated at
10% thresholds calculated once before the loop (`$threshold = max(1, ceil($total / 10))`),
capping DB writes at exactly 10 per job regardless of dataset size. Redis-based progress was
considered but rejected as overkill without a real-time dashboard.

## Decision 8 — Four security guards added to GenerateReportJob

**Context:**
After `GenerateReportJob` was implemented, a dedicated security and improvement analysis was
requested. The job had no protection against path traversal in the file path, no handling for a
deleted task, no guard against concurrent duplicate execution, and no guarantee the PHP stream
resource would be closed on exception.

**Challenge:**
You asked to see the proposed fixes before approving them, then confirmed implementation of the
four highest-priority issues only (items 1–4 from the analysis).

**Outcome:**
Four guards were applied. (1) Path traversal: the integer ID is explicitly cast with `(int)` in
the file path construction — a defensive habit against any future path building that incorporates
user input. (2) Deleted task guard: `Task::find((int) $this->task->id)` is called at the top of
both `handle()` and `failed()`; if the task no longer exists (user deleted, cascade fired), the
method returns early rather than throwing a secondary exception that would mask the original.
(3) Duplicate dispatch guard: after `refresh()`, if `$this->task->status !== TaskStatus::Pending`
the job exits immediately — a second worker processing the same task (from a duplicate dispatch or
a manual Horizon retry) will not overwrite the file or reset progress. (4) Stream resource leak:
`fclose($stream)` was moved into a `finally` block so the stream is always released whether
`handle()` completes normally or throws mid-loop.
