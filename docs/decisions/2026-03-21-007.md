# Session 7 — Bulk Email Sending Feature Architecture

**Branch:** `feature/bulk-email-sending`
**Transcript:** [2026-03-21-007](../sessions/2026-03-21-007.md)

## Decision 1 — `Bus::batch()` with `->allowFailures()` for per-recipient job fanout

**Context:**
The bulk email feature fans out one `SendBulkEmailJob` per recipient. The question was how
to structure the dispatch and what happens when delivery to one recipient fails.

**Challenge:**
An initial plan review identified that without `->allowFailures()`, a single failing job
cancels the entire batch. Jobs already completed have sent their emails; queued-but-not-yet-
started jobs are skipped. The outcome is timing-dependent and unpredictable — some recipients
receive the email, others do not, with no reliable pattern.

**Outcome:**
`Bus::batch()->allowFailures()` was added. Each job runs fully independently: a failure for
one recipient has zero effect on the others. All jobs run to completion or exhaust their own
retries. Only the specific recipient whose job exhausted retries does not receive the email.
The `then()` callback always fires regardless of individual failures, guaranteeing the
delivery report is always generated. This is the decisive difference from all-or-nothing
batch semantics used in file conversion.

## Decision 2 — `failed()` writes only to `delivery_status`; task status owned exclusively by batch callbacks

**Context:**
The initial plan had `SendBulkEmailJob::failed()` setting `task->status = Failed` and storing
an error message. With `allowFailures()`, the `then()` callback still fires after individual
failures and sets `task->status = Completed`. These two writes conflict — `failed()` sets
Failed, then `then()` overwrites it to Completed.

**Challenge:**
The edge case review identified this as a direct contradiction in the plan: `failed()` and
`then()` both wrote to `task->status` with opposing values, in unpredictable order.

**Outcome:**
Responsibilities were split cleanly. `failed()` is responsible only for appending
`$recipient => 'failed'` to `payload['delivery_status']`. It never touches `task->status`.
`task->status` is owned exclusively by the batch callbacks: `then()` sets Completed (always
fires, even with some failures), `catch()` sets Failed (fires only on infrastructure failure
— Redis down, batch record corruption — not on individual job failures). This makes the state
machine unambiguous.

## Decision 3 — No new migration; `tasks.payload` carries all email state

**Context:**
The feature needed storage for subject, body, recipients, an optional attachment path,
`batch_id`, and per-recipient delivery status. The alternatives were a new `email_tasks` table
or the existing `payload` JSON column on `tasks`.

**Challenge:**
The architectural constraint throughout the project is no new migrations for task payload
data. The question was whether the complexity of email state justified an exception.

**Outcome:**
No new migration. The `tasks.payload` JSON column carries all state: `subject`, `body`,
`recipients`, `attachment_path` (null when no PDF uploaded), `batch_id` (written after
dispatch), and `delivery_status` (a map of `email => sent|failed`, accumulated by each job
on completion). This follows the same pattern as the file conversion feature where `batch_id`
and `output_files` live in `payload`. The delivery status map is small (bounded by the
recipient cap), making it suitable for a JSON column.

## Decision 4 — Batch callbacks capture `$taskId` (int), not the full Eloquent model

**Context:**
The `then()` and `catch()` closures in `Bus::batch()` are serialised by Laravel. The initial
plan captured `$task` (the full Eloquent model) inside those closures.

**Challenge:**
The edge case review identified this as a stale model risk: the serialised model snapshot
could be minutes or hours old by the time the batch finishes. Any status or payload changes
made by jobs in the interim would be invisible to the callbacks operating on the stale copy.

**Outcome:**
Batch callbacks capture only `$taskId` (the integer primary key) and call
`Task::find($taskId)` inside the callback to reload a fresh model. This is the correct
pattern for all queue closures that reference Eloquent models — capturing the ID is safe to
serialise; loading fresh inside the callback guarantees current state. A deleted-task guard
(`if (!$task) return`) is applied immediately after the find.

## Decision 5 — Configurable max recipients via `config('mail.bulk_max_recipients', 3)`

**Context:**
The recipient cap needed to be a concrete limit (the request specified 3) but "keep it
abstract so it could change later." Hardcoding 3 in the validation rule would require a code
change to raise the cap.

**Challenge:**
The requirement was explicit: the magic number 3 must not appear anywhere except the
configuration default.

**Outcome:**
A single config key `bulk_max_recipients` was added to `config/mail.php`, defaulting to 3
via `env('MAIL_BULK_MAX_RECIPIENTS', 3)`. `StoreEmailRequest` reads
`config('mail.bulk_max_recipients', 3)` as the `max:` rule value. No other file references
the number. Raising the cap requires only an environment variable change — no code
modifications.

## Decision 6 — `email:rfc` validation only (not `email:rfc,dns`)

**Context:**
`StoreEmailRequest` validates recipient addresses. Laravel's `email:rfc,dns` performs a live
DNS lookup on the domain at request time.

**Challenge:**
The edge case review identified `email:rfc,dns` as a latency risk: DNS resolution for 3
addresses can take 3–15 seconds depending on the resolver. Combined with PHP-FPM request
timeouts and load balancer timeouts, this could cause gateway errors on valid requests.

**Outcome:**
`email:rfc` only. RFC-format validation is synchronous and sub-millisecond. DNS validity is
a deliverability concern, not a format concern — a valid-format address that has no MX record
will fail at SMTP send time, which is already handled by the job's `failed()` hook. Adding
DNS latency to the synchronous HTTP request is the wrong place for that check.

---

## Decision 7 — `distinct` rule added to recipient validation

**Context:**
The initial validation rules allowed `["a@b.com", "a@b.com"]` to pass, which would dispatch
two jobs for the same recipient and send two identical emails.

**Challenge:**
The edge case review identified duplicate recipients as a missing validation rule. The bug
was silent — no error, just an unintended double-send.

**Outcome:**
`'distinct'` was added to the `recipients` array validation. `["a@b.com", "a@b.com"]` now
returns a 422 with a validation error on the `recipients` field. A dedicated feature test
covers this boundary.

---

## Decision 8 — Attachment deleted at the exact stored path, not via wildcard

**Context:**
The `then()` callback deletes the uploaded PDF attachment after all jobs complete, since it
is only needed during sending and should not persist. The initial plan used
`Storage::delete('emails/{task_uuid}/attachment_*.pdf')` — a wildcard pattern.

**Challenge:**
The edge case review noted that `Storage::delete()` does not support glob patterns. The
wildcard call would silently do nothing, leaving the attachment on disk permanently.

**Outcome:**
The attachment path is stored in `payload['attachment_path']` at upload time. The `then()`
callback reads this exact path and calls `Storage::delete($task->payload['attachment_path'])`.
If `attachment_path` is null (no PDF was uploaded), the delete is skipped. This also requires
a null guard on `delivery_status` in the same callback — if all jobs were cancelled before
writing their status, the field may be null. The guard treats null as an empty map and still
generates a valid report CSV.

## Decision 9 — Dedicated `emails` queue for job isolation

**Context:**
All existing jobs run on the default queue. Bulk email jobs fan out to multiple SMTP calls
and can be slow or queue-blocking if an SMTP server is unresponsive.

**Challenge:**
Without queue isolation, a batch of slow email jobs could starve file conversion or report
generation workers.

**Outcome:**
`SendBulkEmailJob` sets `public string $queue = 'emails'`. Email jobs run on a dedicated
queue that can have its own worker(s), backoff configuration, and failure budget, independent
of the default queue. This is a zero-migration, zero-infrastructure change — the `emails`
queue is created on first use by the queue driver.

---

## Decision 10 — Five-layer build order: foundation → mail → job → service → HTTP

**Context:**
The feature spans 12+ files across multiple layers. Building in the wrong order creates
circular dependencies — the service depends on the job, the job depends on the mailable, the
HTTP layer depends on all three.

**Challenge:**
You asked for the implementation to be divided into layers before any code was written.

**Outcome:**
Five layers were defined in strict dependency order:
- **Layer 1** — Config key + `TaskType` enum case. No dependencies; safe to do first.
- **Layer 2** — `BulkEmailMailable` + Blade view. Pure data carrier; no queue or DB.
- **Layer 3** — `SendBulkEmailJob`. Depends on Mailable and the existing Task model.
- **Layer 4** — `TaskService::createBulkEmailTask()`. Depends on all prior layers; the most complex layer (batch wiring, then/catch callbacks, compensating logic).
- **Layer 5** — `StoreEmailRequest`, `EmailController`, route, feature tests. Thin adapter; depends on the complete service.

Each layer is independently testable. Tests for each layer are written alongside the layer, not deferred to the end.
