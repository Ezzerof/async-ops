## Decision 1 — MySQL over SQLite for local development

**Context:**
A local database backend was needed. The two practical options were SQLite (zero setup,
file-based) and MySQL (requires a running server or container). The question was presented
before touching any configuration.

**Challenge:**
You asked for a senior Laravel perspective on which to use. The case for SQLite was
speed of setup. The cases against it were concrete: write locking under concurrent queue
workers, limited JSON column support, and a different SQL dialect that can mask
production bugs (case sensitivity, strict mode differences).

**Outcome:**
MySQL 8 via Docker (`asyncops-mysql`, port 3306). Chosen because the project is
queue-heavy with concurrent DB writes (SQLite has write-locking issues), JSON columns
are used in task payloads (limited SQLite support), and production-identical behaviour
catches real bugs during development that SQLite would mask. You selected this from an
explicit multi-option prompt — the choice was deliberate, not defaulted into.

## Decision 2 — Docker for infrastructure, native PHP for the app

**Context:**
MySQL and Redis needed to run somewhere. Options were native installs, Docker for
everything (including PHP), or a hybrid. The project has Laravel Sail listed as a dev
dependency in `composer.json`, which implied a fully containerised setup.

**Challenge:**
You asked: *"Is it because we must have a PHP container?"* when `localhost:8000` was
unreachable. This surfaced an assumption that Docker meant containerising everything.
The diagnosis revealed that nothing was listening on port 8000 — `php artisan serve` had
never been started. Sail was never initialised; its presence in `composer.json` was
irrelevant.

**Outcome:**
PHP runs natively (already installed at 8.3.6 — no container overhead or Sail setup
needed). MySQL and Redis run in Docker containers (`asyncops-mysql`, `asyncops-redis`).
This gives the portability of containers for stateful services without the friction of
containerising the app itself during development. The hybrid was confirmed explicitly:
"PHP native, databases in Docker. No PHP container needed."

## Decision 3 — `composer run dev` as a single multi-process dev command

**Context:**
The dev stack requires four concurrent processes: PHP server, queue worker, log
tailer, and Vite asset compiler. Running them separately in multiple terminals is
error-prone and easy to forget.

**Challenge:**
No direct challenge — the single-command approach was accepted immediately when
presented as the fix for the unreachable `localhost:8000`. The implicit question was
whether `composer run dev` was the right hook for this, rather than a Makefile or shell
script.

**Outcome:**
`composer run dev` spawns all four processes via `npx concurrently` with named,
colour-coded output (`server`, `queue`, `logs`, `vite`) and `--kill-others` so the
whole stack shuts down cleanly if any process fails. Composer scripts were preferred
over a Makefile because they are already a Laravel convention and require no extra
tooling.

## Decision 4 — Two-folder documentation structure (`sessions/` + `decisions/`)

**Context:**
You wanted to capture the reasoning behind decisions, not just the code. The question
was how to organise that alongside the transcript files. A `docs/` folder was designed
with two sub-folders: `sessions/` for raw transcripts and `decisions/` for extracted
Architecture Decision Records.

**Challenge:**
You asked: *"Will you be able to copy the raw transcript for each session and fill the
decisions?"* — which revealed the intent to use transcripts as source material for
decision records, not just as an archive. The automation model was also discussed: at
end of session, reading the JSONL file and extracting turns into Markdown without any
manual effort.

**Outcome:**
Two folders under `docs/`:
- `sessions/` — verbatim transcripts, one file per conversation, dated and numbered
- `decisions/` — extracted decision records written from the session content

The separation keeps the raw record (high noise, high fidelity) apart from the
distilled record (low noise, durable). Code shows *what* was built; transcripts
capture *why*; decisions make the *why* scannable.

## Decision 5 — Verbatim transcript format over clean summary

**Context:**
When designing the session file template, a choice had to be made: a clean summarised
markdown document or the full verbatim conversation including every tool call and
failed attempt.

**Challenge:**
The automation discussion surfaced the trade-off explicitly. A clean summary is more
readable but lossy — it erases the sequence of attempts, the errors hit, and the
reasoning behind abandoned approaches. Verbatim is noisy but complete and searchable.

**Outcome:**
Verbatim raw. Every command, tool call, error, and retry is preserved. The reasoning:
verbatim transcripts are searchable (`git log -S "keyword"`), preserve the exact
sequence of attempts including failures, and allow post-mortems on *why* an approach
was tried and abandoned — context that a summary would erase. The extraction script
renders tool calls as `● Bash(...)` with truncated output rather than omitting them.

## Decision 6 — Infrastructure and documentation in separate commits

**Context:**
The first commit needed to land infrastructure changes (`package-lock.json`) alongside
documentation (`docs/`). The question was whether to bundle them or separate them.

**Challenge:**
You explicitly rejected the combined approach: *"I want you first to create and push
the infrastructure changes. Do not stage the docs folder."* The plan had originally
grouped them in a single commit.

**Outcome:**
Two separate commits on separate branches:
1. `chore: bootstrap dev environment` — `package-lock.json` only (runnable code)
2. Docs commit — transcripts and decision records (reference material)

Infrastructure and documentation are distinct concerns. Separating them lets reviewers
distinguish between "what we run" and "what explains why", and allows docs to be
updated independently without touching deployable code.
